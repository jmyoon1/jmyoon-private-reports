Autoregressive Entity Retrieval

De Cao et al. · ICLR 2021 (spotlight) · 2020

arXiv:2010.00904 · PDF · SP: SP4

P0Track A

Problem
해결하려는 문제

Entity retrieval was universally framed as multi-class classification over atomic entity labels, matched via a bi-encoder dot product between dense input and entity-metadata embeddings. This design (i) misses fine-grained context–entity interactions without a costly cross-encoder re-ranker, (ii) requires a large and linearly-growing memory footprint to store per-entity dense vectors, (iii) needs hard-negative subsampling to approximate the softmax at training time, and (iv) suffers a cold-start problem for entities lacking descriptions/relations.

원자적 개체 라벨 위의 분류는 메모리 비용이 크고 신규 개체 추가가 어렵다.

Solution / Contribution
해결책 / 기여

Reframe entity retrieval as autoregressive generation of the entity's unique name, conditioned on the mention context, using a fine-tuned BART seq2seq model: $$\text{score}(e|x)=p_\theta(y|x)=\prod_{i=1}^{N} p_\theta(y_i \mid y_{<i}, x)$$ trained with the standard NMT/seq2seq objective (teacher forcing, dropout, label smoothing) — the exact softmax is computed per output token so no negative sampling is needed. At inference, constrained beam search with a prefix trie of valid entity names guarantees outputs are real KB entities; cost is independent of $|E|$.

제약 디코딩으로 개체의 정본 이름을 자기회귀 생성하는 GENRE를 제안하여 약 20배 적은 메모리로 zero-shot 링킹을 달성하며, 신규 개체는 이름 추가만으로 등록된다.

Key Results
주요 결과
  • Document retrieval on KILT: +13.7 precision points on average over prior models (headline claim from abstract/intro).
  • ~20× smaller memory footprint on average vs. competing (bi-encoder) systems.
  • State-of-the-art or very competitive across 20+ datasets on entity disambiguation (InKB micro-F1), end-to-end EL (GERBIL), and page-level document retrieval (R-precision).
  • New entities can be retrieved by only specifying their unambiguous name (no description/relations required).
Concepts Introduced도입된 개념

GENRE (Generative ENtity REtrieval)

  • What it computes: input = mention context $x$ (text with a flagged mention span) → operation = a seq2seq transformer (fine-tuned BART) generates the entity's unique canonical name $y$ token-by-token autoregressively, scoring $\text{score}(e|x)=p(y|x)=\prod_{i=1}^{N} p(y_i|y_{<i},x)$ → output = ranked list of entity names (via beam search).
  • Why it was created: replaces the "classifier over atomic entity labels" (bi-encoder dot-product) paradigm, which (i) misses fine-grained context↔entity interactions, (ii) needs large memory to store dense entity vectors (24GB for 6M Wikipedia pages), (iii) requires hard negative subsampling, and (iv) has a cold-start problem for new entities.
  • Concrete example: input "Superman saved [START] Metropolis [END]" → generates "Metropolis (comics)" as top-1 rather than "Metropolis (1927 film)".
  • How it is used in this paper: the core system, evaluated on 20+ datasets across entity disambiguation, end-to-end entity linking, and document retrieval.

Constrained Beam Search (with prefix trie)

  • What it computes: input = partial generated token prefix → operation = at each decoding step, a prefix tree (trie) of all valid entity names masks the log-probabilities of tokens that would not continue toward a valid entity identifier → output = only valid entity names from the candidate set $E$.
  • Why it was created: unconstrained generation can emit strings that are not valid entity identifiers; the trie forces every beam to end at a real KB entity.
  • Concrete example: BART-tokenized trie over 6M Wikipedia titles = 6M leaves, 17M internal nodes, ~600MB on disk. Beam size $k\approx10$, avg title length 6 BPE tokens; inference cost is independent of $|E|$.
  • How it is used in this paper: inference-time decoding for ED and DR; ensures generated names are in the (possibly candidate-restricted) entity set.

Dynamically markup-constrained decoding (for end-to-end EL)

  • What it computes: input = a document → operation = the decoder copies the source while inserting markup [mention](entity), switching among three states (outside / inside-mention / inside-entity-link); the entity trie is applied only in the inside-entity state, and computed dynamically since the annotated output space is exponentially large → output = document with mentions detected and linked.
  • Why it was created: extends the autoregressive/trie approach from a fixed small output space to the intractable space of span-annotated documents.
  • Concrete example: "In 1503, Leonardo began painting the Mona Lisa." → "In 1503, [Leonardo](Leonardo da Vinci) began painting the [Mona Lisa](Mona Lisa)."
  • How it is used in this paper: the end-to-end EL setting evaluated on the GERBIL benchmark.

5Cs Analysis5Cs 분석

Category: new method (system for entity retrieval/linking) Context: builds on BART (Lewis et al. 2020a) and seq2seq/NMT; contrasts with bi-encoder retrievers BLINK (Wu et al. 2020) and DPR (Karpukhin et al. 2020); evaluated against Le & Titov (2018) for ED and Kolitsas et al. (2018) for EL. Within this survey, central to SP4 (scientific entity/concept linking → matching extracted signals to existing KB nodes vs. spawning new ones). Correctness: — 20+ datasets across 3 task families reported; specific numbers/CIs not yet extracted from experiments section. Contributions: (1) first generative (name-generating) entity retriever; (2) constrained beam search via prefix trie for valid-entity decoding; (3) dynamically markup-constrained decoding for end-to-end EL; (4) ~20× smaller memory footprint; new entities addable by simply appending their name. Clarity: method well-described with equations, figures (trie, markup decoding), and public code; reproducibility appears high.

Implementation Feasibility구현 실현가능성
  • Needs: BART (large) weights; BLINK pretraining data (9M doc-mention-entity triples) + task data; a prefix trie over the target entity name set (~600MB for 6M Wikipedia titles); GPU for fine-tuning; beam search decoding at inference.
  • Code: available at https://github.com/facebookresearch/GENRE (code + pre-trained models).
  • Modifications: build a trie over scientific-KB canonical names; adapt the "link vs. new node" step into an explicit novelty decision; consider replacing fine-tuning with a prompting/constrained-decoding wrapper around an LLM for the SP4 agent setting.
  • Unknowns: missing link — how to construct unique/compositional canonical names for scientific concepts that lack Wikipedia-style titles; exact per-benchmark numbers and hyperparameters not yet read (skim mode).

Condition Delta — transfer to our setting
조건 차이 — 우리 환경으로의 전이

Paper's setting: inputs = English text with a flagged mention (ED) or a raw document (EL) or a query (DR); outputs = ranked Wikipedia entity titles / annotated document; conditions = candidate set = Wikipedia (6M entities) or provided per-mention candidate subsets; requires fine-tuning BART on BLINK (9M triples) + task data; English only. Our setting (SP4, O2): inputs = extracted entity/concept mentions from scientific/academic text; outputs = a link to an existing scientific-KB node or a decision to spawn a new node; conditions = prompting/retrieval or lightweight fine-tuning preferred (not full pretraining); scientific domain KB, not Wikipedia. Differences:

  • Domain: Wikipedia entities vs. scientific concepts/entities — missing link: scientific concepts often lack unique canonical Wikipedia-style titles; a name-trie may not exist for a scientific KB.
  • New-entity handling: GENRE adds entities by appending names to the trie — resolved: partially — this directly maps to SP4's "spawn new node vs. link" decision, but GENRE assumes the name is known/unambiguous, whereas KB construction must decide novelty.
  • Adaptation cost: paper fine-tunes BART on 9M+ triples — reference only: our setting favors prompting/lightweight adaptation; abstract note in all_papers.json says "directly adoptable by a prompting/generation-based agent" (the generate-then-constrain idea is the borrowable pattern, not the fine-tuned weights).
  • Language: English only — reference only for our (likely English scientific text) setting.
