Echoes in the Loop: Diagnosing Risks in LLM-Powered Recommender Systems under Feedback Loops

Park et al. · arXiv · 2026

arXiv:2602.07442 · PDF · SP: SP10

P4Track B

한 줄 요약

LLM 기반 recommender의 feedback loop 위험을 진단한다(2026). 환각(hallucination)된 item/상호작용이 학습 로그에 유입되는 — 실제 item 재가중 루프에는 존재할 수 없는 — 오염 채널을 식별한다.

Problem해결하려는 문제

Prior LLM4RS work evaluates recommendation performance; the systemic risks of embedding LLMs into the recommendation pipeline — popularity bias amplification, hallucinated (non-existent) items, ranking inconsistency — and especially how these risks propagate and accumulate through feedback loops (recommendations fed back as training data for the next cycle) have no systematic, controlled measurement framework. Existing LLM4RS surveys also lack a functional-role perspective that would localize which pipeline stage injects which risk.

Solution / Contribution해결책 / 기여

EchoTrace: a role-aware, phase-wise diagnostic framework on top of a controlled feedback-loop simulation.

  1. Temporal split: interaction log split at cutoff $\tau$ into initial training set $D^{(0)}$ and ground-truth activity set $D_{gt}$ (Eq. 1); $D_{gt}$ preserves realistic activity timing/volume but is never trained on.
  2. Recursive recommend–inject–retrain loop over $T$ periods: active users get Top-$k$ lists ($k$ = actual per-period consumption count), recommended items are injected as observed interactions, model retrains on the accumulated set.
  3. Phase-wise diagnostics: P1 measures bias (attribute-distribution shift vs. ground truth) and hallucination (FEF rate, logical contradictions) in LLM-generated content; P2 measures ranking-level validity, stability, and popularity skew; P3 measures long-term ecosystem effects (embedding separation, exposure polarization).

Headline empirical claim (abstract): LLM components amplify popularity bias, inject spurious signals via hallucination, and produce polarized, self-reinforcing exposure patterns over cycles. [SKIM - verify experimental details]

Concepts Introduced도입된 개념

LLM4RS functional-role taxonomy (5 roles)

  • What it computes: a classification of 77 surveyed LLM-powered-recommender studies (2023–2025, top DM/IR venues) into five roles by where the LLM sits in the pipeline: (R1) LLM-as-Augmenter (generates synthetic user–item interactions), (R2) LLM-as-Representer (generates user/item profiles or embeddings), (R3) LLM-as-Recommender (produces or re-ranks the final list), (R4) LLM-as-XAI (explanations), (R5) LLM-as-RecAgent (agentic orchestration).
  • Why it was created: prior LLM4RS surveys organize by architecture/dataset and lack a functional-role view needed to localize where risks enter the pipeline.
  • Concrete example: LLM-as-Augmenter — prompting an LLM with a user's history plus two cold items and asking which better matches latent preference; the pick is injected as a synthetic training interaction.
  • How it is used in this paper: the diagnostic framework instruments the three dominant roles (Augmenter, Representer, Recommender); XAI and RecAgent are excluded as hard to isolate.

EchoTrace controlled feedback-loop pipeline

  • What it computes: input = a temporally split interaction log $D^{(0)}$ (pre-cutoff train) and $D_{gt}$ (post-cutoff ground truth, used only to set activity timing/volume); operation = for each of $T$ periods: (1) recommend Top-$k$ to users active in $D_{gt}$ that period, with $k$ = the user's actual consumption count; (2) inject recommended items back as observed interactions; (3) retrain the LLM4RS on the accumulated data; output = a fully simulated interaction history where human consumption is progressively replaced by LLM-recommendation-driven consumption.
  • Why it was created: feedback loops in traditional RS are studied, but no controlled, observable setting existed for measuring how LLM-induced biases/hallucinations accumulate over retraining cycles.
  • Concrete example: the counterfactual it operationalizes — "how would the system evolve if users followed only LLM recommendations and those were continuously fed back as training data?"
  • How it is used in this paper: the substrate on which all three diagnostic phases run.

Three-phase risk diagnostics (P1 LLMGC / P2 Ranking / P3 Feedback-loop)

  • What it computes: phase-wise measurements — P1: distributional bias and hallucination in LLM-generated content (profiles, augmented interactions); P2: ranking-level effects (invalid/hallucinated items in lists, ranking instability, popularity skew); P3: accumulation across cycles (embedding separation, polarized exposure across user/item groups).
  • Why it was created: to trace where a risk emerges vs. where it merely manifests, rather than measuring only end-of-pipeline outcomes.
  • Concrete example: FEF rate (factual errors and fabrications) — the proportion of generated attributes/interactions that do not exist in the dataset (e.g., item IDs outside the catalog); plus logical contradictions (LC) — identical inputs yielding inconsistent outputs across repeated runs.   • How it is used in this paper: the per-phase measurement layer answering RQ1–RQ4 (LLMGC-level, ranking-level, feedback-loop-level, ecosystem-level).
Implementation Feasibility구현 가능성
  • Needs: RS benchmark logs with timestamps, an LLM API for the three roles, a backbone recommender (LightGCN), compute for repeated retraining over $T$ periods.
  • Code: available at https://github.com/DongUk-Park/EchoTrace (toolkit release stated in paper; repo liveness unverified [SKIM]).
  • Modifications: to serve our survey's question, the injection rule would need an exposure→click acceptance model instead of full acceptance; metrics for exposure-frequency-vs-preference divergence would need to be added.
  • Unknowns: missing link: which LLMs, datasets, and how many loop periods $T$ were used; effect sizes; whether ranking instability is measured across LLM temperature settings.

Condition Delta — transfer to our setting조건 차이 — 우리 설정으로의 전이

Paper's setting: inputs = user–item interaction logs with timestamps (standard RS benchmarks, e.g., MovieLens-style with attributes); system = LLM4RS (LLM as augmenter/representer/recommender over a backbone like LightGCN); outputs = Top-$k$ ranked lists over closed item catalogs; condition = simulated closed feedback loop where recommendations are injected as consumption. Our setting (SP10, exposure-frequency-as-ranking-proxy survey): concerned with how exposure frequency becomes a self-reinforcing ranking signal — whether frequency of exposure acts as a proxy that ranking systems (including LLM-based ones) mistake for preference/quality. Differences:

  • Full-acceptance user model (every recommended item is injected as consumed) vs. real settings where exposure ≠ consumption: missing link — the paper's loop conflates exposure with interaction, exactly the proxy failure our survey studies, but does not model the gap. [SKIM - verify whether §4 varies acceptance]
  • Diagnosis vs. mitigation: paper measures amplification, offers no debiasing intervention: reference only — useful as measurement instrumentation, not as a solution.
  • LLM4RS-specific risks (hallucinated items) vs. our general exposure-frequency concern: resolved: popularity-bias amplification results transfer directly; hallucination results are LLM-specific extras.
Root-cause vs. Symptom근본 원인 대 증상

Direct — for the problem it poses (no controlled way to observe where and how LLM-induced risks enter and compound), the framework attacks the root cause by instrumenting each pipeline stage inside an explicit closed-loop simulation rather than only scoring end outcomes; it is a diagnostic, though, so it measures rather than mitigates.
