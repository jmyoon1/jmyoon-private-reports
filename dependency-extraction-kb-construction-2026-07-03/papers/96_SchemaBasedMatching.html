A Survey of Schema-based Matching Approaches

Shvaiko, Euzenat · Journal on Data Semantics IV · 2005

PDF · SP: SP7

P1Track A

Problem
해결하려는 문제

By 2005 many schema-matching and ontology-matching systems existed but were described in incompatible vocabularies, developed in three disjoint communities (databases, information systems, AI/knowledge representation). Prior surveys each covered only one community and analyzed disjoint system sets: Rahm & Bernstein [62] covered database schema matching but had no semantic/ontology axis; Kalfoglou & Schorlemmer [39] and Wache et al. [75] covered ontology work but gave no comparative technique-level classification. There was no common conceptual basis to compare techniques across communities or to see which parts of the design space a given system covers — blocking both principled comparison and systematic benchmark design.

매처들이 결과를 boolean/score/typed 등 상이하게 보고하여 비교가 어려웠다.

Solution / Contribution
해결책 / 기여

An empirical finding / framework paper, not a new algorithm. The contribution is a two-tree classification of schema-based (schema-level only, no instance data) elementary matching techniques, built under four stated guidelines — exhaustivity, disjointness, homogeneity, saturation — so that new techniques fall inside existing leaves rather than between categories.

  • Descending tree (Granularity/Input Interpretation): Schema-Based Matching → {Element-level, Structure-level} → {Syntactic, External, Semantic} → basic-technique leaves.
  • Ascending tree (Kind of Input): leaves ← {Terminological (String / Linguistic), Structural (Internal / Relational), Semantic}.
  • The formal I/O of any matcher is fixed as the matching process: $A' = f(o, o', A, p, r)$ where $A$ = input alignment, $p$ = parameters (weights, thresholds), $r$ = external resources (thesauri).
  • The framework is then applied: 8 systems (SF, Artemis, Cupid, COMA, NOM/QOM, Anchor-PROMPT, OLA, S-Match) are located in the space (Table 1), and a system-level tripartite categorization (solutions / theorems / likeness clues) is proposed.

신뢰도 n과 관계 R을 담은 5-tuple ⟨id,e,e',n,R⟩ 정렬 형식과 syntactic/external/semantic 축(Shvaiko–Euzenat)을 제시하여 O2가 필요로 하는 신뢰도+유형 관계 결정 primitive를 공급한다.

Key Results
주요 결과
  • Classifies ~14 basic matcher families across two dual trees; places 8 state-of-the-art systems on the map (Table 1), showing coverage per cell.
  • Empirical meta-finding: matching research up to 2005 concentrated almost entirely on syntactic + external techniques; only S-Match used semantic (model-based, propositional-SAT) techniques — flagged as the promising open frontier because semantic exact algorithms are sound and complete (guarantee discovery of all mappings).
  • Worked numeric anchors used throughout: edit-distance(NKN, Nikon) = 0.4; trigram distance(nkon, nikon) = 1/3; example confidence 0.67 with threshold 0.55.
  • Establishes the classification as a benchmark-design tool (discard one feature at a time per class) — the seed of OAEI evaluation methodology.
Concepts Introduced도입된 개념

Mapping element / Alignment

  • What it computes: input = a pair of entities $(e, e')$ drawn from two schemas/ontologies $o, o'$ → operation = a matcher assigns a confidence $n \in [0,1]$ and a relation $R$ → output = a 5-tuple $\langle id, e, e', n, R\rangle$; the set of all such tuples is an alignment $A'$.
  • Why it was created: prior work reported matches inconsistently (some as [0,1] scores, some as boolean, some as typed relations); the 5-tuple unifies these under one formal object.
  • Concrete example: $\langle id_{5,4},\ \text{Photo\_and\_Cameras},\ \text{Cameras\_and\_Photo},\ 0.67,\ =\rangle$ with threshold 0.55.
  • How it is used in this paper: it is the common output format against which all surveyed systems are compared; relations $R \in \{=, \sqsubseteq, \sqsupseteq, \perp, \sqcap\}$ (equivalence, more/less general, disjointness, overlap).

Granularity/Input-Interpretation classification (element-level vs structure-level × syntactic/external/semantic)

  • What it computes: input = a matching technique → operation = classify it by (i) whether it looks at an entity in isolation (element-level) or in its structural context (structure-level), and (ii) whether it interprets input by pure algorithm (syntactic), via auxiliary resources (external), or via formal semantics (semantic) → output = a leaf class in a two-tree taxonomy (Fig. 3).
  • Why it was created: extends Rahm & Bernstein's [ref 62] schema-matching classification, which lacked the syntactic/external/semantic axis and did not cover ontology-specific (semantic) matchers.
  • Concrete example: edit-distance name matching = element-level / syntactic / string-based; propositional-SAT matching (S-Match) = structure-level / semantic / model-based.
  • How it is used in this paper: the backbone of the survey; every basic technique and every system is placed on this map (Table 1).

Kind-of-Input classification (terminological / structural / semantic)

  • What it computes: input = a technique → operation = categorize by the type of object consumed: strings (terminological), structure (structural), or logical models (semantic) → output = a class in a second, dual tree sharing leaves with the first.
  • Why it was created: gives an ascending ("what is manipulated") reading complementary to the descending ("how it is interpreted") reading.
  • Concrete example: WordNet synonym lookup = terminological/linguistic; children/leaves matcher = structural/relational; DL subsumption = semantic.
  • How it is used in this paper: the two classifications together define the "solution space" that systems are said to cover partially.

Basic elementary matcher families (defined here as the leaves of the taxonomy)

  • Element-level: string-based (prefix, suffix, edit distance, n-gram), language-based (tokenization, lemmatization, elimination), linguistic-resource (WordNet/thesauri), constraint-based (datatype & multiplicity comparison, keys), alignment-reuse, upper-level-ontology-based (SUMO, DOLCE — noted as not-yet-implemented).
  • Structure-level: graph-based (graph matching via fix-point, children, leaves, relations), taxonomy-based (bounded-path matching, super/sub-concept rules), repository-of-structures (metadata similarity), model-based (propositional SAT, DL reasoning).
  • How used: each family is defined with a worked example (e.g., edit distance NKN↔Nikon = 0.4; trigram nkon↔nikon = 1/3) and a citation list of systems using it.

Three system-level categories: alignments as solutions (optimization, e.g. Similarity Flooding, OLA), alignments as theorems (semantics-constrained, e.g. S-Match), alignments as likeness clues (heuristic scores for user selection, e.g. Cupid, COMA, Artemis).

5Cs Analysis5Cs 분석

Category: survey + classification framework (theoretical/organizational, no new system). Context: explicitly extends Rahm & Bernstein 2001 [62]; augmented by Shvaiko 2004 [67] and Euzenat et al. 2004 [28]. Sits above and organizes the concrete matchers of Cupid, COMA, Similarity Flooding, S-Match, OLA, Anchor-PROMPT. Directly underpins the OAEI / Ontology Alignment Contest benchmark line (mentioned as the authors' own follow-up). Correctness: this is a taxonomy, so "correctness" = coverage and disjointness, argued informally via the saturation principle rather than proven. No quantitative evaluation in this paper; comparative claims (e.g. "Cupid performs somewhat better than Dike and Artemis", "COMA dominates ... SF, SemInt") are cited from the original system papers, not independently reproduced. Two leaf classes (upper-level-ontology and DL-based) are admittedly unimplemented at time of writing — speculative. Contributions: (1) the syntactic/external/semantic interpretation axis added on top of [62]; (2) a saturated two-tree taxonomy with worked examples for every leaf; (3) the solutions/theorems/likeness-clues system-level view and Table 1 placement of 8 systems. Clarity: highly reproducible as a conceptual map — definitions and examples are concrete. Not reproducible as an algorithm (there is none).

Contribution vs. Claim기여 대 주장
  • Stated claims: provides a common conceptual basis usable both for analytically comparing existing systems and for designing new ones taking advantage of state-of-the-art; claims the taxonomy is stable (new techniques won't fall between categories) by construction.
  • Demonstrated results: the comparative placement of 8 systems (Table 1) is shown; the classification's usefulness for benchmark design is asserted with reference to the authors' then-nascent OAEI work.
  • Gap: "stability" and "exhaustivity" rest on the informal saturation argument, not proof — and the authors themselves note two leaf classes are unimplemented, i.e. the space is defined partly by extrapolation. No empirical study demonstrates that using the taxonomy actually improves new system design. Comparative performance claims are inherited from cited papers under differing evaluation conditions, not normalized.
Implementation Feasibility구현 실현가능성
  • Needs: nothing to run (survey). To apply the framework: a labeled-graph representation of the two structures, plus off-the-shelf ingredients it catalogs — string metrics (edit distance, n-gram), WordNet/thesaurus, a SAT/DL reasoner for the semantic path.
  • Code: no code (2005 survey). Descendant systems it maps are separately available (COMA/COMA++, S-Match, OLA); the framework itself lives on at www.OntologyMatching.org and seeded OAEI (oaei.inrialpes.fr).
  • Modifications for our setting: (i) add an LLM/embedding matcher as a new leaf under element-level external/semantic; (ii) replace human "likeness clue" consumption with an automated threshold/decision rule for equivalence-vs-new-node; (iii) add noise-robust preprocessing for LLM-extracted labels.
  • Unknowns: missing link — how confidence thresholds should be calibrated for automatic (no-human) merge decisions; missing link — how the semantic (SAT/DL) guarantees survive when axioms come from noisy LLM extraction rather than curated ontologies.
Follow-up Papers관련 후속 논문
  • Rahm & Bernstein, "A survey of approaches to automatic schema matching" (VLDB J., 2001) [ref 62 here] — the parent classification this paper extends; read to see the pre-semantic baseline.
  • Giunchiglia, Shvaiko & Yatskevich, "S-Match: semantic schema matching" (2004) — the sole semantic system here; the model-based/SAT path most relevant to a rigorous equiv/new-node decision.
  • Euzenat & Shvaiko, Ontology Matching (Springer book, 2007/2013) — the authors' book-length successor; the canonical modern reference for SP7.
  • OAEI / Ontology Alignment Evaluation Initiative (2005–) — the benchmark line this classification seeded; needed for evaluation methodology in SP7.
Condition Delta — transfer to our setting
조건 차이 — 우리 환경으로의 전이

Paper's setting: inputs = two hand-authored, schema-level structures (relational/XML/ER schemas or OWL/RDF ontologies), typically small tree/DAG catalogs with clean labels; outputs = an alignment of typed relations with [0,1] confidences; conditions = schema-only (no instance data), largely design-time, human-in-the-loop consumption of "likeness clues"; no learning, no LLMs (pre-2005).

Our setting (SP7 — Ontology and taxonomy alignment for KB construction): inputs = automatically-extracted, possibly noisy concept hierarchies / entity nodes emitted by an LLM-driven dependency-extraction pipeline; outputs = a decision of node equivalence vs. new-node creation when merging into an evolving KB, with a confidence criterion; conditions = large, dynamic, run-time, ideally automatic (agent-driven), 2019–2026 methods including LLM-prompted alignment.

Differences:

  • Manual clean schemas vs. LLM-extracted noisy taxonomies: missing link — the survey's matchers assume well-formed labeled graphs; robustness to extraction noise is out of scope.
  • Design-time, human selects among "likeness clues" vs. run-time autonomous equiv/new-node decision: reference only — the solutions/theorems/likeness-clues taxonomy names the design choice we must make (we need "alignments as solutions/theorems", i.e. a decision, not clues for a human).
  • No embeddings/LLMs vs. LLM-based alignment (SP7's "LLM adaptation" group): missing link — provides the conceptual slot (semantic/external element-level matcher) an LLM matcher would occupy, but no LLM method.
  • Confidence + threshold formulation: resolved / directly reusable — the $\langle id,e,e',n,R\rangle$ + threshold model is exactly the equivalence-vs-new-node confidence criterion SP7 needs; typed relations ($=, \sqsubseteq, \perp$) map onto "equivalent / subsume / distinct → new node".
  • Structure-level taxonomy matchers (bounded-path, super/sub-concept rules, children/leaves): resolved / reference — directly applicable to concept-hierarchy merging in KB construction.
