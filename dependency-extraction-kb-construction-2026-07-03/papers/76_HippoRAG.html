HippoRAG: Neurobiologically Inspired Long-Term Memory for Large Language Models

Jimenez Gutierrez et al. · NeurIPS 2024 · 2024 · 255 citations인용 255회

arXiv:2405.14831 · PDF · SP: SP6

P0Track A

Problem
해결하려는 문제

Current RAG methods encode each passage in isolation, so they cannot integrate knowledge that spans passage boundaries. Real tasks (literature review, legal briefing, medical diagnosis) and even standard multi-hop QA require joining facts across passages. Existing solutions use expensive iterative retrieve+generate loops (IRCoT), and even those fail on "path-finding" multi-hop questions.

LLM의 장기 기억과 다중 홉 검색이 코퍼스 전역 지식 통합에 취약하다.

Solution / Contribution
해결책 / 기여

A RAG framework mimicking the hippocampal memory indexing theory:

  • Offline indexing: LLM ("neocortex") runs OpenIE to build a schema-less KG ("hippocampal index"); dense retrieval encoders ("parahippocampal regions") add synonymy edges.
  • Online retrieval: LLM extracts query named entities → linked to KG query nodes → Personalized PageRank (seeded on query nodes, weighted by node specificity) distributes probability over the graph → node probabilities aggregated over passages via matrix $\mathbf{P}$ to rank passages. This achieves multi-hop reasoning in a single retrieval step.

LLM OpenIE로 스키마 없는 코퍼스 KG를 만들고 Personalized PageRank로 단일 스텝 다중 홉 검색을 수행하는 HippoRAG를 제안한다.

Key Results
주요 결과
  • Multi-hop QA: outperforms SOTA RAG by up to 20 points; ~3 and ~20 point retrieval gains on MuSiQue and 2WikiMultiHopQA.
  • Single-step HippoRAG is 10–30x cheaper and 6–13x faster than iterative IRCoT while comparable/better in performance.
  • Combining HippoRAG with IRCoT yields further gains up to 4% and 20%, plus improvement on HotpotQA.
Concepts Introduced도입된 개념

Schema-less open KG via LLM OpenIE (hippocampal index)

  • What it computes: passages → 1-shot LLM prompting (NER, then triple extraction) → set of noun-phrase nodes $N$ and relation edges $E$ forming an open (schema-free) knowledge graph over the whole corpus. Produces a $|N| \times |P|$ occurrence matrix $\mathbf{P}$.
  • Why it was created: current RAG encodes each passage in isolation, so it cannot integrate knowledge across passage boundaries; a corpus-wide graph provides the associative structure needed for cross-passage (multi-hop) retrieval.
  • Concrete example: (Thomas, researches, Alzheimer's) and (Stanford, employs, Thomas) triples extracted and linked; on MuSiQue the KG has 91,729 nodes, 107,448 triples.
  • How it is used in this paper: serves as the "artificial hippocampal index" that PPR is run over during retrieval.

Personalized PageRank (PPR) for single-step multi-hop retrieval

  • What it computes: query named entities → linked to KG "query nodes" via dense encoder cosine similarity → PPR seeded on those nodes produces a node probability distribution $\vec{n}$ → multiplied by matrix $\mathbf{P}$ to yield per-passage ranking scores.
  • Why it was created: replaces iterative retrieve-then-reason loops (e.g. IRCoT) that are slow/expensive; PPR explores KG paths and identifies relevant subgraphs in a single retrieval step.
  • Concrete example: seeds {Stanford, Alzheimer's} → probability mass flows to Professor Thomas node → passages mentioning Thomas ranked top.
  • How it is used in this paper: the core online retrieval mechanism; PPR = "pattern completion" in the hippocampal analogy.

Node specificity

  • What it computes: local IDF-like signal $s_i = |P_i|^{-1}$ where $P_i$ = set of passages node $i$ was extracted from; each query node's PPR seed probability is multiplied by $s_i$.
  • Why it was created: a neurobiologically plausible, local alternative to global IDF (which would need a global aggregator neuron).
  • Concrete example: "Stanford" (fewer docs) gets larger weight than "Alzheimer's".
  • How it is used in this paper: modulation applied before running PPR to improve retrieval.

Synonymy edges (parahippocampal regions)

  • What it computes: off-the-shelf dense retrieval encoder → adds edges $\tilde{E}$ between distinct noun-phrase nodes whose cosine similarity exceeds threshold $\tau$.
  • Why it was created: connects similar-but-not-identical phrases to aid pattern completion (the OpenIE KG alone under-links synonyms).
  • Concrete example: MuSiQue with ColBERTv2 adds 191,636 synonym edges.
  • How it is used in this paper: augments the KG before PPR.

Category: new method (system/framework) with a mechanism-type (neurobiological) framing Context: builds on OpenIE, Personalized PageRank, and dense retrievers (Contriever, ColBERTv2); positioned against single-step retrievers (BM25/GTR/ColBERTv2/Propositionizer/RAPTOR) and iterative retriever IRCoT. Core reference for RAG-over-KG construction (SP6). Correctness: — abstract/intro claim up to 20% gains; statistical rigor not yet assessed. KG statistics table (Table 1) present and concrete. Contributions: (1) LLM-OpenIE → schema-less open KG as corpus index; (2) PPR-based single-step multi-hop retrieval; (3) node specificity local-IDF signal. Clarity: — method appears reproducible (prompts in appendix, code released); full verification needs deep read.

Contribution vs. Claim기여 대 주장

Abstract claims strong multi-hop gains, large cost/speed advantage, and ability to tackle "path-finding" scenarios out of reach of existing methods. Whether the path-finding claim is empirically validated (vs. illustrated by a case study) requires reading the experiments/case-study sections — intro suggests the path-finding setting is shown via a case study rather than a full benchmark.

Implementation Feasibility구현 실현가능성
  • Needs: instruction-tuned LLM for OpenIE (GPT-3.5-turbo per paper), dense retrieval encoder (Contriever / ColBERTv2), PPR implementation over KG, multi-hop QA corpora.
  • Code: available at https://github.com/OSU-NLP-Group/HippoRAG
  • Modifications: for dependency-extraction KB construction (SP6), would need typed/directed relation extraction and correctness evaluation of the KG itself rather than passage-recall-only tuning.
  • Unknowns: missing link — OpenIE prompt robustness across domains, threshold $\tau$ sensitivity, and KG quality metrics not seen in first 5 pages .

Condition Delta — transfer to our setting
조건 차이 — 우리 환경으로의 전이

Paper's setting: inputs = text passage corpus (multi-hop QA benchmarks MuSiQue, 2Wiki, HotpotQA; ~1000 questions, 6k–12k passages each); outputs = ranked passages for QA; conditions = LLM OpenIE (GPT-3.5), off-the-shelf dense encoders, schema-less open KG, PPR retrieval. Our setting: dependency extraction / KG construction (SP6) — inputs = domain corpus/artifacts; outputs = dependency graph / structured KG; conditions = need explicit relation typing and dependency edges, not just associative retrieval. Differences:

  • Schema-less open KG vs. typed dependency graph: missing link — HippoRAG's open triples are untyped noun-phrase associations; dependency extraction needs typed/directed relations.
  • Retrieval objective vs. graph-construction objective: reference only — HippoRAG optimizes passage recall, not KG correctness/completeness as an end product.
  • OpenIE pipeline for KG construction: resolved / directly borrowable — LLM 2-step NER→triple prompting is a concrete recipe for corpus→KG construction relevant to SP6.
  • PPR for cross-node reasoning: reference only — useful if dependency queries need multi-hop traversal.
