fusion-jena/automatic-KG-creation-with-LLM - ontology + KG construction pipeline

SP: SP6

ToolTrack -

Problem
해결하려는 문제

A semi-automatic pipeline that turns scholarly publication PDFs into an ontology-grounded knowledge graph in four sequential LLM stages: (1) formulate competency questions (CQs) → (2) develop an ontology (TBox) from the CQs → (3) construct the KG (ABox) using that ontology → (4) evaluate the resulting KG (including human evaluation of RAG-generated answers to the CQs). Backs the Kommineni et al. paper; demonstrated on the deep-learning-methodology domain.

LLM으로 온톨로지와 KG를 스키마 우선으로 구성·평가할 필요가 있다.

Solution / Contribution
해결책 / 기여
  • Invocation: python main.py, with all variables/paths set in config.ini. Requires a Hugging Face access token (added to helper_functions.py / env var), a venv from requirements.txt, Python 3.10.16 on Linux.
  • Inputs: scholarly publication PDFs + their metadata.
  • Processing: CQ generation → TBox generation → KG (ABox) population + named-entity recognition → RAG over the corpus to answer CQs → evaluation metrics + human scoring.
  • Outputs: competency questions, ontology (TBox), knowledge graph, NER results, and evaluation statistics.
  • Models compared: Mixtral 8x22B Instruct v0.1, GPT-4o, GPT-3.5, Gemini.

CQ→TBox→KG→평가의 스키마 우선 파이프라인과 human-in-loop CQ 채점을 제공하는 자동 KG 생성 도구로 CQ 수용 테스트 템플릿을 제시한다.

Key Results
주요 결과

351 GitHub stars. Peer-reviewed backing (NLP4KGC @ SEMANTiCS 2024, CEUR-WS Vol. 3874, pp. 19–34; arXiv:2403.08345). Reproducible artifact: code + prompts + data released. From the fusion-jena group (König-Ries lab).

Concepts Introduced도입된 개념

Competency Questions (CQs)

  • What it computes: input = scholarly PDF + metadata → LLM generates a set of natural-language questions the target ontology/KG must be able to answer → output = list of CQs that scope the schema.
  • Why it was created: gives the ontology a grounded, verifiable requirements spec instead of ad-hoc schema design.
  • Concrete example: e.g. "What dataset was used to train the model?" → drives an ontology property usesDataset.
  • How it is used in this paper: Stage 1 of the four-stage pipeline; CQs seed the TBox in Stage 2 and become the human-evaluation targets in Stage 4 (RAG answers the CQs, humans score them).

TBox (terminological box / ontology schema)

  • What it computes: input = CQs → LLM proposes classes + properties → output = ontology (TBox).
  • Why it was created: schema-first grounding so the KG (ABox) has controlled vocabulary rather than free-form triples.
  • Concrete example: classes {Method, Dataset, Metric}, property evaluatedOn.
  • How it is used in this paper: Stage 2 output; constrains Stage 3 KG population.

Semi-automatic ontology+KG pipeline (CQ → TBox → KG → eval)

  • What it computes: four sequential LLM stages turning papers into an evaluated KG with minimal human involvement.
  • Why it was created: manual ontology engineering is expensive; automate it while keeping a human check at evaluation.
  • Concrete example: run main.py, configured via config.ini.
  • How it is used in this paper: the whole tool; compares 4 LLMs across all stages.

Implementation Feasibility구현 실현가능성
  • Needs: Python 3.10.16 (Linux), venv from requirements.txt, Hugging Face token, API access for GPT-4o/GPT-3.5/Gemini; PDFs + metadata as input.
  • Code: available at https://github.com/fusion-jena/automatic-KG-creation-with-LLM (prompts + data included).
  • Modifications: to serve our pipeline, would need (a) a cross-paper dependency-edge target instead of intra-paper ontology facts, (b) entity linking against an existing KB, (c) an incremental add-one-paper loop with DAG-consistency checks.
  • Unknowns: missing link — exact CQ/TBox prompt templates and evaluation metrics not read in skim; Linux-only, single-corpus batch assumptions unverified for streaming use.
Borrow Potential차용 가능성
  • Schema-first grounding: force the agent to derive a controlled vocabulary (TBox) from competency questions before populating edges — reduces free-form triple noise. Adoptable as a pre-extraction step.
  • CQ-driven evaluation loop: use competency questions ("what does a correct dependency link answer?") as the acceptance test for extracted links, with RAG+human scoring. Directly transplantable to validate O1/O2 output.
  • Not adoptable as-is: no incremental/linking/DAG-maintenance machinery — the parts most central to O2/O3.
Follow-up Papers관련 후속 논문
  • Kommineni, König-Ries & Samuel (2024), "Towards the Automation of Knowledge Graph Construction Using Large Language Models" (NLP4KGC @ SEMANTiCS 2024) — the paper this repo implements; read for CQ/TBox prompt and evaluation detail.
  • Kommineni et al. (2024), arXiv:2403.08345, "From human experts to machines: An LLM supported approach to ontology and knowledge graph construction" — v1.0 version with fuller methodology.
Condition Delta — transfer to our setting
조건 차이 — 우리 환경으로의 전이

Tool's setting: inputs = scholarly PDFs + metadata (deep-learning domain); outputs = ontology + intra-corpus KG; conditions = batch run over a fixed corpus, schema-first (CQ→TBox), human eval at the end. Our setting: inputs = a newly-added paper arriving over time; outputs = prerequisite/derivation dependency links to existing KB nodes in a growing DAG; conditions = incremental, agent-driven, prompting/retrieval-first. Differences:

  • Construction target (intra-paper ontology facts vs. inter-paper dependency edges): reference only — different edge semantics.
  • Batch vs. incremental: missing link — no add-one-paper update path.
  • No entity linking against a pre-existing graph: missing link — our O2 core is absent.
  • No DAG-consistency maintenance (cycles, pruning, stale links): missing link — our O3 core is absent.
  • Reusable: the CQ → TBox → populate → evaluate staging and the schema-first grounding discipline: resolved: as a design template for grounding an agent's extraction in a controlled vocabulary before writing links.
