Packed Levitated Marker for Entity and Relation Extraction

Ye et al. · ACL 2022 · 2022 · 143 citations인용 143회

arXiv:2109.06067 · PDF · SP: SP2

P0Track A

Problem
해결하려는 문제

Existing entity/relation extraction methods focus on obtaining better span representations from a pre-trained encoder (T-Concat, solid markers, levitated markers) but process spans / span pairs independently, ignoring the interrelation between spans and between span pairs. Solid markers cannot disambiguate more than two marker pairs in one sequence; the prior levitated-marker approach (PURE Approx.) simply swaps solid for levitated markers for batching efficiency but sacrifices accuracy. Packing many markers naively also blows up PLM complexity quadratically.

관계 추출에서 span 쌍 간 상호관계를 효율적으로 인코딩하기 어렵다.

Solution / Contribution
해결책 / 기여

Pack levitated markers strategically to model span interrelations while keeping near-linear complexity:

  • NER (span classification): neighborhood-oriented packing — sort levitated marker pairs by (start, end) position, split into groups of size ≤ K (K=256), so spans with the same start token are modeled together; span rep $\phi(s_i)=[h_a^{(s)};h_b^{(e)}]$, combined with T-Concat features.
  • RE (span-pair classification): subject-oriented packing — solid markers on the subject, levitated markers on all candidate objects packed into one instance; span-pair rep $\phi(s_i,s_j)=[h_{a-1};h_{b+1};h_c^{(s)};h_d^{(e)}]$. Two-stage cascade (fast NER filter → PL-Marker) to bound inference cost.

이웃/주체 지향 levitated marker를 묶어 span 상호관계를 모델링하는 PL-Marker를 제안하여 SciERC 개체·관계에서 최상급 결과를 달성한다.

Key Results
주요 결과
  • Nested RE: with same BERT_BASE, +1.7% strict F1 on ACE05, +2.5% on ACE04 vs prior; vs PURE (Full) with ALBERT-XXLARGE, +4.1% (ACE05) and +4.3% (ACE04) strict relation F1 with higher speed.
  • SciERC (SciBERT encoder): achieves best performance on both entity and relation extraction (SP2-relevant scientific domain); exact Ent/Rel/Rel+ numbers in Table 3 .
  • NER: +0.1%–1.1% F1 over T-Concat across all six NER benchmarks; +0.4/+0.7/+1.9% over SeqTagger on CoNLL03/OntoNotes5/Few-NERD.
  • Neighborhood-oriented packing beats random packing on all flat NER, +9.4% on Few-NERD (long sentences, 325 candidate spans/sent avg).
Concepts Introduced도입된 개념

Levitated Marker

  • What it computes: input = a text span (start, end) → insert a pair of marker tokens that share position embeddings with the span's start/end tokens (position ids of text tokens unchanged) and are tied to each other by a directional attention mask (visible to partner marker, not to text tokens or other marker pairs) → output = a span representation from the encoder without duplicating the input sequence per span. Introduced by prior work (Zhong & Chen, 2021 / PURE); reused here as the building block.
  • Why it was created: solid markers physically inserted into text cannot handle multiple span pairs in one pass; levitated markers allow batching many spans/pairs in a single forward pass for faster inference.
  • Concrete example: for span "Copenhagen", markers [S Copenhagen][/S] share positions with the span boundary tokens while the sentence tokens keep their original position ids.
  • How it is used in this paper: base mechanism that PL-Marker packs; §3.1 Background.

Packed Levitated Markers (PL-Marker) — the paper's core contribution

  • What it computes: input = candidate spans (or subject + candidate objects) → strategically pack multiple levitated markers into one instance under two schemes → output = span / span-pair representations that model interrelations among co-packed spans.
  • Why it was created: prior marker methods process each span (pair) independently, ignoring interrelation between spans/pairs; and naive packing exacerbates PLM quadratic complexity.
  • Concrete example: for RE, subject "David Green" wrapped in solid markers [S]...[/S], and its candidate objects (his, wife, Dallas) each get a levitated marker pair, all packed into one sequence.
  • How it is used in this paper: the method for both NER (span) and end-to-end RE (span pair).

Neighborhood-oriented Packing

  • What it computes: input = candidate span set → sort span markers by (start position, then end position), split into groups of size ≤ K → output = groups where spans sharing a start token co-occur, letting the model compare neighbor boundaries.
  • Why it was created: to control sequence length (near-linear complexity) while still modeling boundary interrelations; random packing loses this.
  • Concrete example: group size K set to 256 for NER; on Few-NERD (avg 325 candidate spans/sentence) yields +9.4% over random packing.
  • How it is used in this paper: the span-classification (NER) packing strategy, §3.2.

Subject-oriented Packing

  • What it computes: input = a subject span + its candidate object spans → emphasize subject with solid markers, pack all candidate objects with levitated markers into one instance (a fusion scheme) → output = span-pair representation $\phi(s_i,s_j)=[h_{a-1};h_{b+1};h_c^{(s)};h_d^{(e)}]$ modeling same-subject pairs jointly.
  • Why it was created: two levitated marker pairs cannot both use directional attention without losing partner identification; fusion of solid (subject) + levitated (objects) resolves this while modeling all same-subject pairs together.
  • Concrete example: subject "David Green" in [S]/[/S], objects his/wife/Dallas each in [O]/[/O], one forward pass predicts all three relations.
  • How it is used in this paper: the span-pair (RE) packing strategy, §3.3; includes an inverse object→subject relation for bidirectional prediction and an auxiliary object-entity-type loss.

5Cs Analysis5Cs 분석

Category: new method (span/span-pair representation for IE) Context: builds directly on levitated markers from PURE (Zhong & Chen, 2021); compares against DyGIE++, PURE (Full/Approx.), T-Concat, SeqTagger. Related to span pre-training (SpanBERT) and knowledge-infusion lines. Correctness: 5 seeds, average reported, std in appendix — reasonable rigor. Full verification . Contributions: (1) first to apply levitated markers to NER; (2) neighborhood-oriented + subject-oriented packing schemes that model span (pair) interrelation; (3) SOTA RE with higher speed via two-stage cascade. Clarity: method described with equations and figures; code released. Reproducibility .

Contribution vs. Claim기여 대 주장

— abstract claims advances on "six NER benchmarks" and 4.1–4.3% RE gains on ACE04/05 plus best SciERC. ACE gains and NER deltas are stated with numbers in intro; distinguishing claimed generality from the exact per-dataset SciERC demonstration requires reading the experiments/tables (Table 3) in full.

Implementation Feasibility구현 실현가능성
  • Needs: pre-trained encoders (BERT-base, ALBERT-xxlarge, SciBERT, RoBERTa-large), the labeled IE datasets, A100-class GPU for speed parity; two-stage NER→RE pipeline.
  • Code: available at https://github.com/thunlp/PL-Marker
  • Modifications: for open KB construction, the fixed relation-type classifier and gold-entity-type auxiliary loss would need adaptation to the target schema.
  • Unknowns: missing link: exact SciERC Ent/Rel/Rel+ figures and whether packing group size / span-length hyperparameters transfer to our documents .

Condition Delta — transfer to our setting
조건 차이 — 우리 환경으로의 전이

Paper's setting: inputs = sentences (with cross-sentence context expanded to 512 tokens) from ACE04/05, SciERC, CoNLL03, OntoNotes5, Few-NERD; outputs = entity spans + typed relations; conditions = supervised, encoders BERT/ALBERT/RoBERTa/SciBERT; nested + flat NER; end-to-end RE via two-stage cascade. Our setting: SP2 — dependency/relation extraction for KB construction, likely scientific-document entity-relation extraction (SciERC is the directly relevant benchmark). Differences:

  • Domain: paper covers general + scientific (SciERC/SciBERT) — SciERC is directly on-target for scientific KB construction: resolved: SciERC + SciBERT results reported.
  • Task framing: supervised span/pair classification with gold entity types vs open KB triple extraction: missing link — degree of overlap with our KB schema unclear .
  • Output granularity: fixed relation-type inventory vs open/dependency relations: reference only .
