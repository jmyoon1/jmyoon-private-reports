A Survey of Graph Retrieval-Augmented Generation for Customized Large Language Models

Zhang et al. · arXiv · 2025 · 117 citations인용 117회

arXiv:2501.13958 · PDF · SP: SP6

P1Track B

Problem
해결하려는 문제

The field of Graph-based RAG has grown rapidly but prior surveys (Peng et al. 2024, Procko & Ochoa 2024, Han et al. 2024) offered only a flat workflow description (indexing → retrieval → generation) without a usage-based taxonomy, without a systematic component-level treatment of retrieval/integration techniques, and without a practitioner-oriented catalog of open-source projects, benchmarks, and domain applications. There is no organized reference explaining how to build and retrieve over graph-structured knowledge to customize LLMs for specialized domains, nor a consolidated view of open challenges (knowledge quality, conflict, privacy, efficiency).

맞춤형 LLM을 위한 graph RAG 방법이 파편화되어 있다.

Solution / Contribution
해결책 / 기여

Survey (no new algorithm). Contributions:

  1. A three-category taxonomy (Knowledge-based / Index-based / Hybrid GraphRAG) crossed with a three-stage pipeline (Knowledge Organization §IV, Knowledge Retrieval §V, Knowledge Integration §VI).
  2. Retrieval techniques classified into six families — string-matching, semantic-similarity, logical-reasoning, GNN-based, LLM-based, RL-based — plus strategies (multi-round, post-retrieval, hybrid). Tabulated per-system (Table I) by query-side input, graph-side input, matching, and pruning method.
  3. Integration classified into fine-tuning (node/path/subgraph-level, Table II) vs. in-context learning (graph-enhanced CoT, collaborative KG refinement, Table III).
  4. Appendix catalog: benchmark datasets (WebQSP, CWQ, MetaQA, KQAPro, GraphRAG-Bench, UltraDomain, etc.), domain applications (Table IV), and open-source projects (Microsoft GraphRAG, Nano-GraphRAG, LightRAG, Fast GraphRAG, Medical GraphRAG).

Knowledge/Index/Hybrid taxonomy와 OIE 기반 코퍼스→KG 절을 정리하여 SP6의 추출+링킹 하위 파이프라인을 매핑한다.

Key Results
주요 결과
  • Reports GraphRAG generates LLM responses with 26%–97% fewer tokens than traditional RAG (cited, breadth of Fig. 4 comparison).
  • Companion resource: Awesome-GraphRAG GitHub repo (github.com/DEEP-PolyU/Awesome-GraphRAG). 117 citations as of extraction.
  • Catalogs 40+ representative systems across three categories and ~25 benchmark datasets; no single headline metric (survey).
Concepts Introduced도입된 개념

GraphRAG (Graph Retrieval-Augmented Generation)

  • What it computes: input = user query + external corpus/KG → operation = organize knowledge as a graph (nodes = entities/concepts/text-chunks/topics, edges = dependencies/semantic relations), retrieve query-relevant subgraph or reasoning path via graph traversal, then integrate into LLM prompt or fine-tuning → output = grounded LLM answer. Formally defined as "a subclass of RAG that leverages graph structure to organize and retrieve knowledge," replacing the flat vector database of naive RAG with a structural database.
  • Why it was created: naive (flat-text) RAG retrieves only semantically-similar chunks, missing multi-hop intermediate concepts, sacrificing cross-document contextual links via chunking, and scaling poorly. Graph structure preserves entity relationships and hierarchies.
  • Concrete example: for "connection between concept A and D," flat RAG retrieves only A- and D-related chunks; GraphRAG traverses A→B→C→D path. Reported 26%–97% fewer generation tokens vs. traditional methods.
  • How it is used in this paper: it is the central object the entire survey taxonomizes across three stages (organization, retrieval, integration).

Three-category GraphRAG taxonomy (the survey's own contribution)

  • What it computes: classifies every GraphRAG system by how it uses the graph → (1) Knowledge-based (graph as knowledge carrier: explicit KG of entities/relations), (2) Index-based (graph as index over raw text chunks/topic nodes), (3) Hybrid (both — nodes carry key info and link to source chunks).
  • Why it was created: prior surveys (Peng et al. [27], Han et al. [30]) only described a flat workflow (indexing→retrieval→generation); this taxonomy adds a usage-based axis.
  • Concrete example: Microsoft GraphRAG [24] (community-summary KG) = knowledge-based; KGP [46] (TF-IDF/entity index graph over passages) = index-based; CodexGraph [77] (code graph both carrying dependency structure and indexing source snippets) = hybrid.
  • How it is used in this paper: top-level organizing structure of §IV (Fig. 3 taxonomy tree).

Knowledge Graph Construction from Corpus (via OIE) — most relevant to this survey's SP

  • What it computes: input = unstructured text corpus → operation = Open Information Extraction (traditional TextRunner/Stanford-OIE [127–130] or LLM-based OIE [131,132]) to extract relational triples; optionally coreference resolution, node/edge merging by TF-IDF overlap, community summarization → output = attributed KG.
  • Why it was created: many domains lack high-quality pre-built KGs; must build the KB from raw documents (directly the dependency-extraction + KB-construction problem).
  • Concrete example: GraphReader [62] summarizes chunks into atomic facts and extracts key elements (nouns/verbs/adjectives) as nodes; GraphCoder [47] builds a Code Context Graph capturing control-flow and data-dependency edges between code statements.
  • How it is used in this paper: §IV.B.1 — one of two directions under "Graphs as Knowledge Carriers."

Reasoning path / subgraph retrieval

  • What it computes: input = query + KG → operation = generate relation paths as plans (RoG [65]), beam search over triples (ToG [21]), RL agent sampling (KnowGPT [5]), or lightweight MLP subgraph extractor (SubgraphRAG [69]) → output = reasoning chain / concise subgraph fed to LLM. Extracting an ideal subgraph is stated to be NP-hard.
  • Why it was created: KG incompleteness causes low recall for direct path lookup; large KGs need pruning to fit LLM context.
  • Concrete example: ToG beam-searches most-probable relational triples; SubgraphRAG's MLP allows tunable subgraph size for an efficiency-effectiveness trade-off.
  • How it is used in this paper: §V retrieval-technique taxonomy (similarity / logical / GNN / LLM / RL-based retrievers).
5Cs Analysis5Cs 분석

Category: survey. Context: Positions itself against three prior GraphRAG surveys ([27] Peng, [29] Procko/Ochoa, [30] Han) and builds on the KG+LLM roadmap [125]. Anchors on Microsoft GraphRAG [24], LightRAG [25], ToG [21], RoG [65], HippoRAG [64]. Correctness: — as a survey it reports no independent experiments; the "26%–97% fewer tokens" and per-dataset statistics are cited from primary works, not re-validated. No error bars (not applicable). Taxonomy boundaries (esp. index vs. hybrid) are somewhat fuzzy. Contributions: (i) usage-based 3-category taxonomy; (ii) component-level retrieval/integration technique tables; (iii) practitioner resource catalog (companion repo Awesome-GraphRAG). Clarity: Clear high-level structure; individual methods described at survey depth (not reproducible from text alone).

Contribution vs. Claim기여 대 주장

— The paper claims a "more systematic and comprehensive" review than prior surveys and "actionable insights ... for practitioners to deploy GraphRAG in production." What it demonstrates is the taxonomy, technique tables, and resource catalog. The production-deployment and cost claims (e.g., GPT-4 pilot customer service "thousands of dollars," fine-tuning "hundreds of GB") are asserted illustratively without a controlled study. Being a survey, there is no empirical validation of any efficiency/quality claim — all numbers are inherited from cited primary works. Gap to check in deep read: whether the three-category taxonomy is exhaustive/mutually-exclusive for systems not listed.

Implementation Feasibility구현 실현가능성
  • Needs: nothing to "run" (survey). To adopt its methods for SP6: an OIE/LLM triple extractor, a graph store (Neo4j per Nano-GraphRAG), an embedding model + vector DB (Milvus/Faiss), and an LLM for community summarization / retrieval.
  • Code: companion resource list at https://github.com/DEEP-PolyU/Awesome-GraphRAG; primary open-source pipelines — Microsoft GraphRAG, Nano-GraphRAG (~1,100 LOC), LightRAG, Fast GraphRAG, Medical GraphRAG.
  • Modifications: for scientific-literature dependency KB, adapt §IV.B.1 corpus→KG construction to paper-level nodes and citation/derivation edges; replace QA benchmarks with extraction P/R/F1 evaluation.
  • Unknowns: missing link — survey gives no guidance on evaluating KB construction accuracy (vs. retrieval accuracy) or on scientific-paper-specific ontology design.
Follow-up Papers관련 후속 논문
  • Microsoft GraphRAG — Edge et al. 2024 "From local to global: A graph RAG approach to query-focused summarization" [24] — the foundational community-summary KG pipeline; core reference for SP6 architecture.
  • LightRAG — Guo et al. 2024 [25] — dual-level (entity + global-key) index construction; a lightweight KB-construction pattern.
  • StructGraphRAG — Zhu et al. 2024 [78] & GraphFusion — Yang et al. 2025 [224] — scientific-research/document KG construction, closest to SP6's scientific-literature target.
  • GraphRAG-Bench — Xiang et al. 2025 [232] — tailored benchmark spanning construction→retrieval→generation; candidate evaluation harness.
  • "When to use graphs in RAG" — Xiang et al. 2025 [232-adjacent, arXiv:2506.05690] — analysis of when graph structure actually helps; relevant scoping evidence.
Condition Delta — transfer to our setting
조건 차이 — 우리 환경으로의 전이

Paper's setting: inputs = user query + external corpus or pre-built KG (Freebase, Wikidata, DBpedia, SPOKE, UMLS, domain corpora); outputs = grounded LLM answers for domain QA / dialog / summarization; conditions = general survey scope across QA-centric benchmarks, code, biomedical, legal, education domains. Our setting (SP6 — LLM-agent & RAG pipelines for scientific-literature KB construction): inputs = scientific/academic papers (citations, methods, results); outputs = a knowledge base of inter-paper dependencies (derivation lineage, method-use, comparison) built end-to-end by an LLM agent; conditions = extraction + linking + graph maintenance via prompting/retrieval or lightweight fine-tuning. Differences:

  • Domain (general/biomedical/code QA vs. scientific-literature dependency KB): reference only — no scientific-paper-lineage dataset in the survey; StructGraphRAG [78]/GraphFusion [224] touch scientific-research KG construction (closest analogs, worth follow-up).
  • Task framing (retrieval-for-QA vs. KB construction as the end goal): resolved: §IV.B.1 "KG Construction from Corpus" (OIE-based triple extraction, coreference, node merging) is exactly the extraction+linking sub-pipeline SP6 needs.
  • Graph as index vs. graph as the deliverable KB: missing link — survey optimizes graphs for downstream retrieval quality, not for KB completeness/accuracy metrics (P/R/F1) that SP6 evaluates on.
  • Automation architecture (LLM-agent end-to-end): resolved: partially — LLM-based retrievers (§V.B.4), CogMG [112] collaborative KG refinement, and open-source Microsoft/Nano-GraphRAG pipelines provide reusable agent patterns.
