KARMA: Leveraging Multi-Agent LLMs for Automated Knowledge Graph Enrichment

NeurIPS 2025 · 2025 · 29 citations인용 29회

arXiv:2502.06472 · PDF · SP: SP6

P0Track B

Problem
해결하려는 문제

Manually curating scientific KGs cannot scale with >7M articles/year. Conventional NLP extractors need expensive supervised tuning (3–5k labeled examples per relation type) and fail to generalize beyond a fixed schema; single-agent LLM extractors hallucinate on complex relations, cannot hold schema consistency across documents, and scale quadratically on full text. There is no verification mechanism when extracting open-schema knowledge and integrating it into a heterogeneous existing KG.

단일 LLM 추출기는 복잡 관계에서 환각하고 문서 간 스키마 일관성을 유지하지 못한다.

Solution / Contribution
해결책 / 기여

New system/pipeline (system contribution). Decompose KG enrichment into nine cooperating LLM agents orchestrated by a Central Controller across three phases:

  • Parsing: Ingestion (normalize PDF/HTML, extract metadata) → Reader (segment + relevance-score $R(s_j)$, drop if $< \delta$) → Summarizer (condense to high-signal $u_j$, preserve entities/numbers).
  • Extraction: Entity Extraction (LLM-NER + ontology filter + embedding normalization, spawn new nodes past threshold $\rho$) → Relationship Extraction (multi-label $p(r|\hat{e}_i,\hat{e}_j,u_j)$, keep $r$ with prob $\geq \theta_r$) → Schema Alignment (map new entity/relation to KG types via $\arg\max_\tau \text{LLM}_{SAA}$, else flag for ontology expansion).
  • Integration: Conflict Resolution (LLM debate → discard/review/integrate) → Evaluator (Confidence/Clarity/Relevance logistic gate, mean $\geq \Theta$) → KG Storage.

Core objective: $G_{new}=G \cup \bigcup_{i=1}^{n}\text{Extract}(p_i)$, adding only novel triplets, each verified before integration.

9개 협력 에이전트로 추출→링킹→검증을 분해한 KARMA를 제안하며, newness 임계값 ρ로 신규 노드 생성을 결정하고 conflict-resolution·evaluator gate로 정확도를 4.6–14.4% 높인다.

Key Results
주요 결과
  • DeepSeek-v3 backbone identifies up to 38,230 new entities (genomics coverage gain $\Delta_{Cov}$) at 83.1% LLM-verified correctness ($R_{LC}$).
  • Conflict resolution reduces conflict edges; automatic evaluation + conflict resolution improve LLM-based accuracy by 4.6%–14.4%.
  • Beats GLM-4 single-agent baseline across domains; DeepSeek-v3 best on 17/24 (71%) metrics.
  • Ablation: removing Summarizer drops CQA 22.9% (0.612→0.472, genomics); removing Conflict Resolution drops correctness (0.831→0.790); removing Evaluator drops RLC 9.7% (0.668→0.603, metabolomics) — each agent contributes.
  • Genomics yields 53.1 entities/article vs. 14.6 for metabolomics (3.6× coverage gap).
Concepts Introduced도입된 개념

KARMA multi-agent enrichment pipeline (nine specialized LLM agents)

  • What it computes: input = existing KG $G=(V,E)$ + corpus of unstructured documents $P=\{p_1,...,p_n\}$; operation = a hierarchical pipeline of nine LLM agents under a Central Controller (Ingestion → Reader → Summarizer → Entity Extraction → Relationship Extraction → Schema Alignment → Conflict Resolution → Evaluator) that parses each document, extracts candidate triplets, verifies them, and integrates survivors; output = augmented graph $G_{new}=G \cup \bigcup_i \text{Extract}(p_i)$ containing only novel triplets $t \notin E$.
  • Why it was created: single-agent / monolithic LLM extraction suffers from (1) hallucination on complex relations, (2) inability to maintain schema consistency across documents, (3) quadratic cost on full-text articles. Decomposing into specialized agents enables cross-agent verification and linear-time processing.
  • Concrete example: from a genomics abstract, the pipeline emits MTX-531 --treats--> HNSCC (Conf 0.99, Rel 0.80, Clr 0.50) after entity normalization, schema typing, conflict check, and evaluator scoring.
  • How it is used in this paper: it is the entire method; every result in Tables 1–2 is produced by running this pipeline over 1,200 PubMed articles with GLM-4 / GPT-4o / DeepSeek-v3 as interchangeable backbones.

Entity normalization via joint-embedding distance + newness threshold

  • What it computes: input = raw entity mention $e$; operation = $\hat{e}=\arg\min_{v\in V} d(\phi(e),\psi(v))$ over KG entities, where $\phi,\psi$ map mention and KG entity into a shared BERT-style embedding space; output = canonical entity, OR flag as new and add to candidate set $V^+$ if $\min_v d > \rho$.
  • Why it was created: to deduplicate mentions ("acetylsalicylic acid" → "Aspirin") and, crucially for incremental construction, to decide when a mention warrants spawning a new node rather than linking to an existing one.
  • Concrete example: threshold $\rho$ separates a match to existing Aspirin (MESH:D001241) from creating a new vertex.
  • How it is used in this paper: it is the linking/spawn-decision step of the Entity Extraction Agent — directly analogous to the survey's "link to existing node or spawn new node" requirement (O2).

Conflict Resolution Agent (LLM-debate contradiction check)

  • What it computes: input = new triplet $t=(\hat{e}_h,r,\hat{e}_t)$ + a candidate conflicting triplet $t'$ in $G$; operation = LLM debate prompt classifying {Agree, Contradict, Ambiguous} and choosing action {discard, review, integrate}; output = keep / drop / escalate-to-human decision.
  • Why it was created: naive LLM extraction lets contradictory edges (e.g., DrugD treats DiseaseE vs. existing DrugD exacerbates DiseaseE) accumulate, degrading graph consistency.
  • Concrete example: two high-confidence contradictory triplets → decision "Contradict", action "review".
  • How it is used in this paper: ablation shows removing it drops correctness (CQA 0.831→0.790 in genomics); it is the graph-consistency-maintenance component (O3 analogue).

Evaluator Agent triple gating (Confidence / Clarity / Relevance)

  • What it computes: input = per-triplet verification signals; operation = three logistic aggregations $C(t)=\sigma(\sum\alpha_i v_i)$, $Cl(t)=\sigma(\sum\beta_j c_j)$, $R(t)=\sigma(\sum\gamma_k r_k)$; integrate iff mean $\geq \Theta$; output = binary integration decision.
  • Why it was created: to filter low-quality/low-relevance triplets before they pollute the KG in the absence of a gold standard.
  • Concrete example: integrate(t)=1 if (C+Cl+R)/3 ≥ Θ.
  • How it is used in this paper: final gate before KG storage; ablation shows removal most degrades usability.

5Cs Analysis5Cs 분석

Category: system / engineering (multi-agent LLM pipeline). Context: Builds on AutoGen-style multi-agent frameworks and multi-agent debate (Liang et al. 2023) for verification; positions against three KG-construction generations (WordNet/ConceptNet rules → BioBERT/SapBERT neural → LLM open-schema). Closest paper in this survey to end-to-end agentic incremental KB construction (SP6 O1–O3). Correctness: No gold-standard benchmark; evaluation is LLM-self-judged (RLC via hold-out LLM) plus limited human eval (2 experts, 0–1 scale) — the paper's own Limitations section flags reliance on LLM-based metrics rather than expert validation. No error bars/CIs reported. Ablations are single-run. Contributions: (1) modular nine-agent architecture enabling cross-agent verification; (2) domain-adaptive prompting for diverse scientific fields; (3) LLM-based verification (conflict + evaluator) mitigating hallucination and schema drift. Clarity: High — each agent has an equation and a full example prompt (Appendix 8), input/output JSON schemas given; reproducible in principle from the description.

Contribution vs. Claim기여 대 주장

— Read full PDF this pass, so preliminary distinction:

  • Stated claims: automated, scalable, accurate KG enrichment; cross-agent verification mitigates hallucination and schema inconsistency; modular/extensible for dynamic updates.
  • Demonstrated: on 1,200 PubMed biomedical articles (genomics/proteomics/metabolomics), coverage and LLM-judged correctness gains over a single-agent baseline; ablations confirm each agent helps.
  • Gap: "accuracy" is largely LLM-self-judged (RLC = hold-out LLM voting), not gold-standard or broad expert validation (human eval = 2 experts, 0–1 scores). Evaluated only in biomedical domain despite general-framework framing. No comparison to non-LLM KG-construction baselines or to other agentic systems (only vs. its own single-agent variant). "Incremental" is demonstrated as batch enrichment of an existing KG, not streaming continuous updates with DAG-consistency guarantees.
Implementation Feasibility구현 실현가능성
  • Needs: an existing seed KG with typed schema; a domain ontology for entity normalization (UMLS/MeSH here); one LLM backbone served via API (GLM-4 / GPT-4o / DeepSeek-v3 all work); embedding model (BERT-style) for entity normalization; prompt templates for all nine agents (fully given in Appendix 8).
  • Code: — not linked in the read pages; abstract/intro give no GitHub URL. Prompts are reproducible from Appendix.
  • Modifications for our setting: replace biomedical relation ontology with prerequisite/citation-dependency relation types; add a DAG-maintenance agent (cycle detection, transitive-reduction, stale-link detection) that KARMA lacks; substitute UMLS-style normalization with a CS/ML concept linker (retrieval over KB nodes); redefine Evaluator relevance for dependency plausibility.
  • Unknowns: missing link — how the Central Controller schedules/parallelizes agents (orchestration details); missing link — how $\rho, \delta, \theta_r, \Theta$ thresholds are calibrated per domain; missing link — whether integration is truly incremental/streaming vs. batch.

Condition Delta — transfer to our setting
조건 차이 — 우리 환경으로의 전이

Paper's setting: inputs = biomedical PubMed PDFs + existing biomedical KG with typed schema (Drug/Disease/Gene/Protein); outputs = new (head, relation, tail) triplets integrated into KG; conditions = domain ontologies (UMLS/MeSH/SNOMED) available for normalization, relation set is biomedical (treats/causes/inhibits), verification by LLM debate + score gating.

Our setting: inputs = a newly-added scientific paper (CS/ML) + growing knowledge base of concept/topic nodes; outputs = prerequisite and citation-lineage dependency links ("Requires: [topic] (+ novel idea X)"); conditions = agent-driven via prompting/retrieval, must keep a global dependency DAG consistent (cycle avoidance, redundant-edge pruning, stale-link detection).

Differences:

  • Domain (biomedical → CS/ML papers): reference only — the agent architecture is domain-agnostic; prompts are domain-adaptive by design.
  • Node/edge semantics (biomedical entity triplets → concept-prerequisite + citation-derivation dependencies): missing link — KARMA extracts factual entity relations, not prerequisite/derivation dependencies between research concepts; the relation ontology and "Requires" semantics must be re-specified.
  • Graph type (heterogeneous KG, no acyclicity constraint → strict DAG): missing link — KARMA's Conflict Resolution checks logical contradiction, not structural cycles/redundancy; no DAG-maintenance (cycle avoidance, transitive-edge pruning) mechanism.
  • Linking/spawn decision (embedding-distance threshold $\rho$ → link-to-existing-node vs. spawn-new-node): resolved/directly reusable — this is exactly the "link or spawn" step the survey needs (O2), transferable via prompting + embedding retrieval.
  • Ontology dependency (UMLS/MeSH available → no ready ontology for CS/ML concepts): missing link — Schema Alignment relies on a typed ontology; our target lacks an equivalent canonical concept ontology.
  • Verification (LLM-debate conflict + score gate → same, but for dependency correctness): resolved/pattern reusable — the confidence/clarity/relevance gate and debate-based conflict resolution transfer conceptually.
