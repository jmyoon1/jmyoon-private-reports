Knowledge Graphs

Hogan et al. · ACM Computing Surveys 2021 · 2021

arXiv:2003.02320 · PDF · SP: SP8

P1Track A

Problem
해결하려는 문제

Prior to this survey, "knowledge graph" lacked a unifying technical account: the literature covered narrow slices (embeddings, IE, ontologies, graph databases) in isolation, with no comprehensive map of how graphs are built, enriched, assessed, refined, and published, nor how these relate to existing data-management topics. For SP8 specifically: there was no single authoritative taxonomy of the KB-construction pipeline (extraction → integration → quality → refinement) with consistent terminology.

추출된 KG는 불완전·중복·모순되어 사용 전 품질을 측정해야 한다.

Solution / Contribution
해결책 / 기여

Empirical/organizational contribution (survey), not a new method. It:

  1. Fixes an inclusive KG definition and contrasts graph data models (directed edge-labelled,

heterogeneous, property graphs, graph datasets) and query primitives.

  1. Structures schema/identity/context (semantic/validating/emergent schema; PIDs; reification).
  2. Separates deductive (ontologies, rules, DLs) from inductive (analytics, embeddings, GNNs, symbolic

learning) knowledge.

  1. Provides the construction/enrichment taxonomy (§6): human collaboration; text IE

(pre-proc→NER→EL→RE); markup extraction (wrappers, web tables, deep web); structured mapping (direct/custom R2RML, tree mappings, KG-to-KG); ontology engineering + ontology learning.

  1. Defines quality dimensions + metrics (§7) and refinement tasks (§8).
  2. Covers publication (FAIR, Linked Data, access protocols, usage control) and real-world open +

enterprise KGs (§10).

정본 구성 파이프라인과 consistency/validity 품질 차원, completion+correction refinement(minimal hitting set 수리)을 정리한 서베이로, 유지·일관성 목표를 명명하는 어휘를 제공한다.

Key Results
주요 결과

Survey — no benchmark numbers. Headline organizing facts:

  • Four-stage text-to-KG IE pipeline (pre-processing → NER → EL → RE) is the canonical construction

path from unstructured sources.

  • Quality decomposed into 4 clusters / ~11 dimensions (Accuracy, Coverage, Coherency, Succinctness).
  • Refinement = completion (link prediction) + correction (fact validation, inconsistency repair).
  • Empirically grounds the taxonomy against major open KGs: DBpedia, YAGO, Freebase, Wikidata (>3B

edges when Freebase froze in 2015), BabelNet, NELL (~120M edges), OpenCyc.

Concepts Introduced도입된 개념

Authoritative tutorial-survey (37 pp. in ACM CSUR; 135 pp. with appendices) defining the entire KG landscape. Only concepts directly relevant to SP8 (dependency extraction / KB construction) are expanded; the rest are named for reference.

Knowledge graph (working definition)

  • What it computes: n/a (definitional). "A graph of data intended to accumulate and convey

knowledge of the real world, whose nodes represent entities of interest and whose edges represent relations between these entities." Data graph may be a directed edge-labelled (RDF-style) graph or a property graph.

  • Why it was created: prior definitions of "knowledge graph" were contentious/inconsistent

(Appendix A surveys ~4 categories); this inclusive definition anchors the survey.

  • Concrete example: "Santiago is the capital of Chile" as an edge; "all capitals are cities" as a

quantified (ontological) statement requiring rules/ontologies.

  • How it is used: the umbrella under which construction, enrichment, assessment, refinement,

publication are organized.

KG creation & enrichment pipeline (§6) — the core of SP8 relevance

  • What it computes: legacy sources (text, markup, tables/trees, other KGs) → graph-structured

data, via extraction or mapping. Text path uses a 4-stage IE pipeline: Pre-processing (tokenisation, POS, dependency parsing, WSD) → Named Entity Recognition (NER) → Entity Linking (EL) → Relation Extraction (RE, binary + n-ary; closed vs. Open IE).

  • Why it was created: KGs must be assembled from diverse, incomplete, large-scale sources; no

single method suffices.

  • Concrete example: sentence "Santiago has flights to Rapa Nui, which was named a World Heritage

Site in 1995" → NER + EL maps Rapa Nui→Easter Island; RE yields edge Santiago —flight→ Easter Island and an n-ary reified named / World Heritage Site / 1995.

  • How it is used: §6 is the section SP8 most depends on — it is the canonical taxonomy of how

edges (dependencies/relations) get extracted and integrated into a KB.

Quality assessment dimensions (§7)

  • What it computes: qualitative dimensions + quantitative metrics. Groups (after Batini &

Scannapieco): Accuracy (syntactic, semantic, timeliness); Coverage (completeness, representativeness/bias); Coherency (consistency = free of logical contradictions; validity = free of shape/constraint violations); Succinctness (conciseness, representational-conciseness, understandability).

  • Why it was created: extracted KGs are incomplete, duplicated, contradictory; "fitness for

purpose" must be measured before use.

  • Concrete example: shape City constrained to ≤1 country; edges `Chile ←country— Santiago

—country→ Cuba` yield a validity violation.

  • How it is used: supplies the evaluation standards (consistency/validation dimensions) that SP8

can cite as the vocabulary for assessing an extracted KB's quality.

KG refinement (§8): completion (link prediction — general/type/identity links) and correction (fact validation against external sources; inconsistency repair via minimal hitting sets).

Named-only (standard-to-experts, not expanded): RDF/RDFS/OWL, SHACL/ShEx shapes, Description Logics (ALC→SROIQ), reification/n-ary/RDF*, quotient-graph emergent schema, KG embeddings (TransE/DistMult/ComplEx/RotatE/RESCAL/TuckER…), GNNs (RecGNN/ConvGNN), rule mining (AMIE, DRUM, NeuralLP), FAIR + Linked Data publication principles.

5Cs Analysis5Cs 분석

Category: survey / tutorial. Context: builds on and positions itself against prior tertiary literature (Paulheim 2017 KG refinement survey; Wang et al. 2017 embeddings; Ji et al. 2020; Pan et al. 2017 book; Fensel et al. 2020). Co-authored by a large consortium from a Dagstuhl seminar. Correctness: definitional/taxonomic; claims are illustrative rather than experimental. Formal appendices (B) give precise semantics for models, queries, shapes, DLs, embeddings, GNNs, rule mining. No experiments/error bars — not applicable to a survey. Contributions (SP8-relevant): (i) the canonical KB-construction/enrichment pipeline taxonomy; (ii) the quality-assessment dimension/metric framework including consistency & validity; (iii) the completion/correction refinement taxonomy. Clarity: exceptionally clear; single running example (Chilean tourism) threads all sections. Highly reproducible as a reference map (not as a reimplementable method).

Contribution vs. Claim기여 대 주장

As a survey the claim is comprehensiveness ("a general unifying summary"). Table 1 substantiates breadth by showing this tutorial covers ~22 topics in-depth vs. narrower coverage in prior surveys. The gap detectable only on close reading: coverage breadth trades off against depth — the paper itself notes several dedicated surveys give "more in-depth technical details on their chosen topic," so for any single SP8 sub-task (e.g., dependency-based relation extraction) one must follow the pointers rather than rely on this survey alone.

Implementation Feasibility구현 실현가능성
  • Needs: nothing to run (survey). To use for SP8: pick the relevant subsection (§6.2 for text

IE, §7 for quality metrics, §8 for refinement) and follow citations to concrete tools.

  • Code: not applicable; open-access PDF at arXiv:2003.02320 (v6, matches ACM CSUR 54(4)).
  • Modifications: none; use as a framing/vocabulary reference and citation hub.
  • Unknowns: missing link — the survey names dozens of extraction techniques but this skim did

not extract the specific dependency-parse-based RE methods SP8 needs; a deep pass on §6.2.4 (RE) and its citations [24, 356, 446, 487, 582] is warranted if SP8 centers on relation/dependency extraction.

Condition Delta — transfer to our setting
조건 차이 — 우리 환경으로의 전이

Paper's setting: inputs = heterogeneous legacy sources (text, HTML/markup, relational/CSV tables, tree formats, other KGs) + human experts; outputs = a graph-structured KB (RDF or property graph) plus schema/ontology; conditions = web-scale, incomplete data, OWA-by-default, multi-source. Our setting (SP8, inferred): dependency extraction for KB construction — extracting typed relations/dependencies from sources and assembling/validating them into a knowledge base, likely at smaller/domain scale with emphasis on the extraction step and downstream quality. Differences:

  • Scope: survey covers the whole lifecycle; SP8 focuses on extraction+construction → reference only

for the non-extraction stages; resolved: §6.2 (IE pipeline) + §7 (quality) directly usable as the framing taxonomy.

  • Granularity: survey stays at the level of naming techniques, not implementing them → **missing link:

concrete algorithm/tooling for a specific dependency-extraction method** must come from cited primary papers (e.g., OpenIE, distant supervision, dependency-parse-based RE).

  • Terminology: survey's "consistency/validity" quality dimensions → **resolved: adopt as the

evaluation standard** for extracted-KB correctness (this is the abstract's stated value for SP8).

  • Modality: survey is modality-broad (text, tables, markup); SP8 emphasis likely text/dependency →

reference only for the non-text mapping paths.
