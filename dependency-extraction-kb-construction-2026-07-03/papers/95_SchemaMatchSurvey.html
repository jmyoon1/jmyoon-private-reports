A survey of approaches to automatic schema matching

Rahm, Bernstein · VLDB Journal · 2001

SP: SP7

P1Track A

Problem
해결하려는 문제

Prior schema-matching work consisted of individually designed systems (Cupid, LSD, SemInt, DIKE, ARTEMIS, etc.) with no shared conceptual framework. There was no common vocabulary or classification to describe what a matcher exploits (schema structure vs data instances, element names vs constraints), how matchers are combined, or how to compare approaches — making it hard to reuse ideas across data integration, warehousing, and message-mapping applications.

스키마 매칭 접근을 분류할 기초 어휘가 필요하였다.

Solution / Contribution
해결책 / 기여

Empirical/conceptual survey. The contribution is a taxonomy, not an algorithm:

  • Distinguish individual matcher criteria along orthogonal dimensions:
  • Schema-level vs instance-level (uses schema metadata vs actual data values).
  • Element-level vs structure-level (matches atomic elements vs whole substructures).
  • Language-based vs constraint-based (names/descriptions vs keys, datatypes, cardinalities).
  • Matching cardinality (1:1, 1:n, n:1, n:m).
  • Auxiliary information (dictionaries, thesauri, global schemas, reuse of prior match results, user feedback).
  • Distinguish combining matchers: hybrid (multiple criteria in one algorithm) vs composite (aggregate independent matchers' results).
  • Survey and classify existing prototypes within this framework and identify open problems (reuse of matches, evaluation methodology).

schema/instance × element/structure × linguistic/constraint 축의 기초 taxonomy(Rahm–Bernstein)를 제시하여 매처 설계·분류의 어휘를 제공한다.

Key Results
주요 결과
  • No benchmark numbers — the deliverable is the taxonomy itself and the classification of ~10 existing prototype systems into it.
  • Establishes the schema-level/instance-level × element-level/structure-level × linguistic/constraint matrix as the standard descriptive framework.
  • Identifies match reuse and lack of standard evaluation as open problems.
Concepts Introduced도입된 개념

Schema matching

  • What it computes: input = two schemas S1, S2 (each a set of elements: tables, attributes, XML nodes) → operation = compute semantic correspondences (a "match" / mapping) between elements → output = a set of mapping elements, each relating one or more elements of S1 to one or more of S2, optionally with a similarity/confidence value and a mapping expression.
  • Why it was created: data integration, warehousing, and e-commerce message translation required reconciling independently designed schemas; doing this by hand is slow and error-prone, so the paper frames the problem to enable automatic solutions.
  • Concrete example: matching Customer.CName in one schema to Client.Name in another as a 1:1 correspondence with high similarity.
  • How it is used in this paper: the central object the whole taxonomy classifies.

Taxonomy of matching approaches (the survey's core contribution)

  • What it computes: a classification tree that partitions matcher algorithms along orthogonal axes → schema-level vs instance-level; element-level vs structure-level; language/linguistic-based vs constraint-based; matching cardinality; and use of auxiliary information (dictionaries, thesauri, prior matches, user input).
  • Why it was created: prior matching systems were described ad hoc with no common vocabulary, making them impossible to compare.
  • Concrete example: a linguistic name-similarity matcher = schema-level + element-level + language-based; a matcher using column value distributions = instance-level + element-level + constraint-based.
  • How it is used in this paper: every surveyed system (Cupid, LSD, SemInt, DIKE, ARTEMIS, TranSCM) is placed in this taxonomy.

Hybrid vs composite matchers

  • What it computes: hybrid = a single algorithm that combines multiple match criteria (e.g. name + datatype) internally; composite = independently executed matchers whose individual results are combined by an aggregation/selection strategy.
  • Why it was created: no single criterion is reliable alone; combination strategy determines accuracy and flexibility.
  • Concrete example: composite system runs a name matcher and a structure matcher separately, then combines their similarity matrices via a weighted average before selecting best matches above a threshold.
  • How it is used in this paper: defines the design space for combining individual matchers into full systems.

5Cs Analysis5Cs 분석

Category: survey Context: foundational reference underpinning later ontology-alignment surveys (Shvaiko & Euzenat) and KB-integration work; establishes vocabulary reused across SP7's schema/ontology-matching thread. Correctness: conceptual survey — no experiments of its own; correctness = coverage and coherence of the taxonomy, which became the field-standard framing. Contributions: (1) the schema-matching taxonomy; (2) individual vs combined (hybrid/composite) matcher distinction; (3) survey and classification of existing prototypes plus open-problem list. Clarity: taxonomy is well-defined and reproducible as a classification scheme.

Contribution vs. Claim기여 대 주장

— As a survey, it claims to organize the design space of schema matching; distinguishing stated framing from demonstrated coverage requires reading the full classification of each surveyed system.

Implementation Feasibility구현 실현가능성
  • Needs: nothing to run — conceptual framework; usable as a checklist to classify/design matchers.
  • Code: not available (survey, paywalled at Springer; no direct PDF).
  • Modifications: to apply to KB construction, extend the instance-level/auxiliary-information branches to cover embedding- and LLM-based matchers.
  • Unknowns: missing link — full-text behind Springer auth; per-system classification details not read in skim.
Follow-up Papers관련 후속 논문
  • A survey of schema-based matching approaches (Shvaiko & Euzenat, 2005) — extends this taxonomy to ontology matching; the natural SP7 successor.
  • Ontology Matching (Euzenat & Shvaiko, book) — consolidates alignment methods built on this vocabulary.
  • Generic Schema Matching with Cupid (Madhavan, Bernstein, Rahm, 2001) — the hybrid matcher classified in this survey.
Condition Delta — transfer to our setting
조건 차이 — 우리 환경으로의 전이

Paper's setting: inputs = pairs of database/XML schemas; outputs = element-to-element correspondences (mappings); conditions = 2001-era relational/XML data integration, no ML embeddings, matchers are rule/heuristic/linguistic-based. Our setting: dependency extraction for KB construction — inputs = heterogeneous schemas/ontologies/data sources whose elements must be aligned to build/merge a knowledge base; outputs = correspondences feeding dependency/relation extraction. Differences:

  • Schema pairs vs open-domain KB elements: reference only — taxonomy transfers as vocabulary but the paper predates ontology/KG alignment.
  • Heuristic/linguistic matchers vs modern embedding/LLM matchers: reference only — modern methods slot into the same schema-level/instance-level, element/structure axes.
  • No learned representations: missing link — the taxonomy's "auxiliary information + instance-level" branch is where learning-based methods would extend it.
