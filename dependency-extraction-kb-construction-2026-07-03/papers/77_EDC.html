Extract, Define, Canonicalize: An LLM-based Framework for Knowledge Graph Construction

Zhang & Soh · EMNLP 2024 · 2024 · 136 citations인용 136회

arXiv:2404.03868 · PDF · SP: SP6

P0Track A

Problem
해결하려는 문제

Prior LLM-based KGC methods require the KG schema (entity/relation types) inside the extraction prompt to produce valid triplets. Large, complex schemas (e.g. Wikipedia) exceed the context window or get ignored by the LLM, so these methods work only on small domain-specific datasets. Additionally, no method handles the case where no predefined schema exists but a high-quality, succinct self-generated schema is still desired.

스키마를 프롬프트에 넣는 LLM-KGC는 위키피디아 규모 스키마에서 문맥 한계로 실패한다.

Solution / Contribution
해결책 / 기여

Three-phase framework, no LLM fine-tuning of the base model:

  1. Extract — few-shot open IE prompt produces schema-free triplets.
  2. Define — LLM writes a natural-language definition per induced schema component (used as semantically rich side info).
  3. Canonicalize — embed definitions with a sentence transformer; vector-similarity search retrieves candidate schema components; LLM verifies/selects the replacement. Target Alignment drops non-mappable components; Self Canonicalization adds them to a growing schema.

Optional +R (Refinement): a trained Schema Retriever (fine-tuned E5-mistral-7b, InfoNCE loss) retrieves relevant schema relations; these plus prior extractions form a "hint" for a second EDC pass.

스키마 없이 먼저 추출하고 각 관계를 정의한 뒤 임베딩 유사도+LLM 검증으로 정규화하는 Extract-Define-Canonicalize를 제안하여 fine-tuning 없이 200개 이상 관계로 확장한다.

Key Results
주요 결과
  • EDC matches or outperforms SOTA (REGEN on WebNLG; GenIE on REBEL, Wiki-NRE) in Partial-F1 across all three datasets in Target Alignment, with no parameter tuning.
  • GPT-4 is the best OIE backbone; GPT-3.5-turbo and Mistral-7b comparable.
  • EDC+R (single refinement iteration with Schema Retriever hint) consistently and significantly improves over plain EDC for all tested LLMs.
  • Handles schemas up to 200 relation types (REBEL) — far larger than prior in-prompt-schema methods.
Concepts Introduced도입된 개념

EDC (Extract-Define-Canonicalize) framework

  • What it computes: input text → (1) open IE prompt yields free-form [subject, relation, object] triplets with no schema constraint → (2) LLM writes a natural-language definition for each induced relation/entity type → (3) canonicalization maps each component to a target or self-generated schema via vector similarity search + LLM verification → output canonical KG triplets.
  • Why it was created: prior LLM-KGC methods (ChatIE, CodeKGC) must embed the full KG schema in the prompt to guarantee valid triplets; large real-world schemas (Wikipedia-scale) exceed the LLM context window or get ignored. EDC removes the schema from the extraction prompt and defers schema conformance to a post-hoc step.
  • Concrete example: from "Alan Shepard … was a member of the Apollo 14 crew", Phase 1 extracts [Alan Shepard, participatedIn, Apollo 14]; Phase 2 defines participatedIn; Phase 3 finds target-schema relation mission is the closest match and, after LLM confirms, rewrites to [Alan Shepard, mission, Apollo 14].
  • How it is used in this paper: it is the core method, evaluated on WebNLG, REBEL, Wiki-NRE.

Schema Canonicalization (Target Alignment vs. Self Canonicalization)

  • What it computes: for each open relation, embed its LLM-generated definition with a sentence transformer, retrieve top-k nearest schema components by cosine similarity, present them as multiple-choice to an LLM that decides the correct replacement (or "None"). Target Alignment maps to a fixed schema (non-mappable components are dropped); Self Canonicalization builds a schema from scratch (non-mappable components are added, expanding it).
  • Why it was created: clustering-based canonicalization (CESI) over-generalizes — e.g. merging "is brother of", "is son of", "was professor of" into one cluster. EDC uses LLM verification on top of similarity to reject bad merges.
  • Concrete example: canonicalization choices A. mission B. season … F. None-of-above, chosen by LLM from vector-search candidates.
  • How it is used in this paper: Phase 3 of EDC; the two modes are the two evaluation settings.

Schema Retriever (EDC+R refinement)

  • What it computes: a fine-tuned E5-mistral-7b-instruct embedding model that projects input text and schema components into a space where cosine similarity captures "is this schema component present in this text" (relevance, not semantic equivalence). Trained with InfoNCE on (text, defined-relation) positive pairs. Retrieved relations + previously extracted entities form a "hint" fed back into the extraction prompt for a second EDC pass.
  • Why it was created: to scale EDC to large schemas (RAG-style) and bootstrap extraction from the previous iteration, recovering obscure schema relations the OIE misses.
  • Concrete example: refinement adds [Alan Shepard, selectedByNasa, 1959] — an obscure target-schema relation missed in the first pass.
  • How it is used in this paper: the EDC+R variant; shown to consistently improve over plain EDC.

5Cs Analysis5Cs 분석

Category: new method (system/framework) — LLM-based KG construction pipeline. Context: builds on open IE (Li et al. 2023), RAG (Lewis et al. 2020), and canonicalization work (CESI/Vashishth 2018, Putri 2019). Positioned against LLM-KGC methods ChatIE (Wei 2023) and CodeKGC (Bi 2024), and seq2seq SOTA REGEN (WebNLG) and GenIE (REBEL, Wiki-NRE). Correctness: — reports both automatic (token-based WebNLG script P/R/F1, named-entity metrics) and manual evaluation; notes reference triplets in REBEL/Wiki-NRE are non-exhaustive, penalizing valid EDC extractions (honest limitation). Error bars not confirmed in skim. Contributions: (1) EDC flexible framework handling both large predefined and self-generated schemas without fine-tuning; (2) trained Schema Retriever for RAG-style schema retrieval; (3) empirical evidence on 3 KGC benchmarks. Clarity: prompts are given verbatim; code released. Reproducible from description.

Contribution vs. Claim기여 대 주장

— claims "higher-quality KGs vs SOTA with significantly larger schemas"; abstract asserts flexibility across predefined and no-schema settings. Whether the automatic F1 gains (vs the acknowledged non-exhaustive references on REBEL/Wiki-NRE) fully substantiate the "higher quality" claim needs the experiments/manual-eval section. The paper itself flags that valid EDC triplets are penalized by incomplete gold sets, so the manual evaluation is the load-bearing evidence — confirm on deep read.

Implementation Feasibility구현 실현가능성
  • Needs: an LLM for OIE + define + canonicalize (GPT-4/3.5-turbo or local Mistral-7b), a sentence-transformer for definition embeddings, E5-mistral-7b for similarity search; for +R a fine-tuned E5-mistral-7b Schema Retriever (InfoNCE, text–relation pairs). GPU for local models.
  • Code: available at https://github.com/clear-nus/edc
  • Modifications: re-author prompts and target schema for the dependency/technical domain; retrain or evaluate zero-shot transfer of the Schema Retriever; possibly add entity-linking beyond relation canonicalization.
  • Unknowns: missing link — exact F1 deltas (in figures/appendix, not in-text on skim); missing link — retriever transferability to a new domain without labeled pairs.
Follow-up Papers관련 후속 논문
  • CodeKGC (Bi et al., 2024) — in-prompt-schema baseline EDC circumvents; contrast for schema-scaling limits.
  • CESI (Vashishth et al., 2018) — clustering-based canonicalization baseline; the over-generalization failure EDC's LLM-verification targets.

Condition Delta — transfer to our setting
조건 차이 — 우리 환경으로의 전이

Paper's setting: inputs = free text (sentences/short passages from WebNLG, REBEL, Wiki-NRE); outputs = canonicalized [subject, relation, object] triplets conforming to a target or self-generated relation schema (45–200 relation types); conditions = general-domain / Wikipedia-style text, English, off-the-shelf LLMs (GPT-4/3.5, Mistral-7b) with no base-model fine-tuning; one trained retriever. Our setting: dependency extraction + KB construction from (likely) software/technical or domain artifacts; the target system does extraction + linking + schema-mapping — architecturally the same extract→link→canonicalize spine. Differences:

  • Domain: general-encyclopedic text vs. our dependency/technical domain — reference only (method is domain-agnostic; prompts/schema would need re-authoring).
  • Schema source: EDC canonicalizes to a target relation schema or self-generates one → directly maps to our schema-canonicalization step. resolved: EDC's Target Alignment + LLM-verified similarity search is the exact pattern for KB schema canonicalization.
  • Retriever training: Schema Retriever needs (text, relation) training pairs → missing link: our setting may lack labeled schema-retrieval pairs; unclear if the retriever transfers zero-shot.
