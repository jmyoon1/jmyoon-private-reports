Agent-OM: Leveraging LLM Agents for Ontology Matching

Qiang, Wang, Taylor, et al. · VLDB 2025 · 2023

arXiv:2312.00326 · PDF · SP: SP7

P0Track B

Problem
해결하려는 문제

Ontology matching has two entrenched paradigms — knowledge-based expert systems (LogMap, AML; precise but labour-intensive) and ML/predictive systems (BERTMap; need large training sets). Neither transfers to LLMs: LLMs are too large to retrain, domain ontologies have only ~100–200 entities (far below the ~1000 needed to fine-tune GPT-3), and web-only LLMs risk leaking sensitive data. Prior LLM-for-OM work (He 2023, OLaLa, LLMs4OM) reduces matching to one-shot binary classification prompts, which are quadratic in cost, hit token limits, hallucinate, and are unstable across runs. No prior work builds an autonomous LLM-agent infrastructure (planning + tools + memory) for OM.

모든 쌍에 대한 이진 LLM 프롬프트는 2차 복잡도로 토큰 한계·환각에 취약하다.

Solution / Contribution
해결책 / 기여

An agent-powered design paradigm: two Siamese LLM agents sharing memory.

  • Retrieval Agent ($Agent\_R$): CoT planning invokes tools — Metadata Retriever ($R_{int}$: category/type), Syntactic Retriever (tokenise+normalise only — no stopword removal/stemming, per authors' prior study), Lexical Retriever ($R_{ext}$: "What is the meaning of {entity}? Context:{context}" + rdfs:comment), Semantic Retriever ($R_{int}$: verbalise triples), Hybrid DB Store (relational for metadata + vector DB for embedded content).
  • Matching Agent ($Agent\_M$): Hybrid DB Search → Metadata Matcher (exact type filter) + Syntactic/Lexical/Semantic Matcher (cosine similarity $S_C(A,B)=\frac{A\cdot B}{\|A\|\|B\|}$, threshold 0.90, top@k 3) → Matching Summariser (RRF fuse) → Matching Validator (LLM self-check) → Matching Merger (bidirectional intersection).
  • Runs the whole pipeline twice (once per ontology as source), merges. Implemented in LangChain tool-calling agents over 10 LLMs (4 commercial API, 6 open-source 7–9B via Ollama), PostgreSQL+pgvector.

각 개체를 한 번 임베딩해 vector DB에 저장하고 top-k 유사도 검색+임계값으로 매칭하는 Agent-OM을 제안하여, 복잡도를 N_s×N_t에서 (k+1)(N_s+N_t)로 낮추고 학습 없이 prompting만으로 온톨로지 매칭을 수행한다.

Key Results
주요 결과
  • OAEI Conference track: F1 ranks 3/13 (2022), 5/12 (2023) among ~15 systems — "above average", close to best on simple tasks.
  • OAEI Anatomy track, trivial correspondences: F1 2nd-highest in both 2022 and 2023.
  • OAEI Anatomy track, non-trivial correspondences: F1 superior to 11 systems incl. LLM-based OLaLa; only behind DL-based Matcha (which benefited from a large training set).
  • OAEI MSE track: best F1 on Test Case 1 and Test Case 2 (beats all 4 baselines); Test Case 3 competitive, bettered by Matcha in 2023.
  • Ablation: search-based Agent-OM far cheaper than LLM-with-Context ($2.7\times10^5$ vs $4.7\times10^6$ tokens) at comparable/better precision; validator and merger each improve precision + F1 with slight recall drop.
  • "Moravec's paradox": strong on complex/few-shot tasks but not outstanding on simple tasks.
Concepts Introduced도입된 개념

Agent-OM framework (Siamese retrieval + matching agents)

  • What it computes: input = source ontology $O_s$ + target ontology $O_t$ → two autonomous LLM agents (Retrieval Agent, Matching Agent) each with own CoT planning module + tool set but shared memory → output = alignment $A$ = set of equivalent entity pairs $\{(e_i,e_j)\}$. Reformulates classical OM pipeline $R_{int}\Rightarrow R_{ext}\Rightarrow M_{sel}\Rightarrow M_{alg}\Rightarrow M_{ref}$ as $Agent\_R(R_{int}\Rightarrow R_{ext}\Rightarrow R_{sto})\Rightarrow Agent\_M(M_{sea}\Rightarrow M_{sel}\Rightarrow M_{alg}\Rightarrow M_{ref})$.
  • Why it was created: one-shot binary "is A ≡ B?" LLM prompting over all entity pairs has $N_s\times N_t$ complexity, hits token limits, and is unstable/hallucination-prone.
  • Concrete example: matching cmt#ProgramCommitteeChair → confOf#Chair_PC on the CMT-ConfOf conference alignment.
  • How it is used: the entire paper is the framework + a proof-of-concept implementation (LangChain tool-calling agents, PostgreSQL + pgvector hybrid DB, 10 LLMs).

Search-based OM via hybrid database (vs. binary-comparison OM)

  • What it computes: retrieve each entity's syntactic/lexical/semantic info once, embed it (OpenAI text-embedding-ada-002, dim 1536), store in a vector DB; matching = cosine-similarity search + relational-DB metadata filter instead of pairwise LLM prompts. Complexity drops from $N_s\times N_t$ to $(k+1)(N_s+N_t)$.
  • Why it was created: bypass token limits and quadratic prompting cost of naive LLM OM.
  • Concrete example: similarity_threshold = 0.90, top@k = 3; validated inequality $(k+1)(N_s+N_t)<N_s\times N_t$ when $N_s,N_t \gg k+1$.
  • How it is used: core of the Matching Agent; ablation 6.1.4.

Reciprocal Rank Fusion (RRF) for syntactic/lexical/semantic fusion

  • What it computes: $RRF(d)=\sum_{r\in R}\frac{1}{k+r(d)}$ with $k=0$; fuses three ranked lists (syntactic, lexical, semantic matchers) into one ranking. Borrowed from Cormack et al. 2009, not novel here.
  • Why it was created (in this paper): mean-score fusion cannot break ties between near-identical entities and is destroyed by a single missing (e.g. semantic) matcher; RRF handles both.
  • Concrete example: fused list [(1.0, [Chair_PC, Member_PC]), (0.5, [Scholar]), (0.33, [Author])].
  • How it is used: Matching Summariser tool ($M_{alg}$).

Matching Validator + Matching Merger (hallucination-reduction tools)

  • What it computes: Validator = self-check binary prompt ("Is X equivalent to Y? Context:... yes/no") applied top-down over RRF ranks (à la SelfCheckGPT); Merger = keep only correspondences found bidirectionally ($O_s\Rightarrow O_t$ AND $O_s\Leftarrow O_t$).
  • Why it was created: reduce false-positive mappings (non-existent + counter-intuitive).
  • Concrete example: Validator returns "yes" for ProgramCommitteeChair≡Chair_PC, "no" for ≡Member_PC.
  • How it is used: refinement stage $M_{ref}$; ablations 6.1.5–6.1.6 show precision/F1 gains with slight recall loss.

Unified naming convention (Type-1 name vs. Type-2 code)

  • What it computes: if an entity name is a natural-language string (Type 1, e.g. ProgramCommitteeChair) use it directly; if it is an opaque code (Type 2, e.g. NCI_C33736) substitute its rdfs:label/rdfs:comment, and disambiguate reused names with serial-Category-Type-Name IDs.
  • Why it was created: LLMs mis-synthesise meanings of biomedical codes (e.g. both MA_0000270 and NCI_C33736 → "liver", wrongly).
  • How it is used: preprocessing before retrieval; matters for the Anatomy track.

5Cs Analysis5Cs 분석

Category: system/engineering (new design paradigm + proof-of-concept implementation). Context: builds on classical OM (LogMap, AML, FCA-Map), ML OM (BERTMap, LogMap-ML), early LLM OM (OLaLa [ref], LLMs4OM, He 2023), and LLM-agent scaffolding (AutoGPT, BabyAGI, CoT, ICL, RAG). Within SP7 it is the Empirical / most agent-native node, adjacent to BERTMap, LLMs4OM, LLMOracle. Correctness: — results reported for single best model (gpt-4o) over a single run; authors note LLM non-determinism causes "slight" but "not significant" differences; no error bars/CIs. Similarity threshold framed loosely as a "confidence interval". Claims of significance for validator/merger are hedged (not statistically significant at their chosen hyperparameters). Contributions: (1) first LLM-agent design paradigm for OM; (2) search-based OM with hybrid DB cutting $N_s\times N_t$ → $(k+1)(N_s+N_t)$; (3) empirical gains on complex/few-shot OAEI tasks. Clarity: high — pseudo-tool-call tables (Tables 2–3), exact SQL, exact prompts, full reproducibility appendix (hardware, pip versions, config). Reproducible from description + public GitHub.

Contribution vs. Claim기여 대 주장

— Abstract claims "results very close to best long-standing performance on simple OM tasks and significant improvement on complex and few-shot tasks", and the conclusion escalates to "Agent-OM is all you need" and "100% accurate, fully-automated, domain-independent OM within reach". Skim-level flags to confirm in deep read: (a) simple-task results are only "above average" (rank 3–5/13), not SOTA, and the paper concedes this via Moravec's paradox; (b) headline numbers are single-run, single-model (gpt-4o) with no CIs; (c) evaluation is TBox-only (classes + properties, equivalence relation only) — ABox/instance matching, subsumption, and complex relations are explicitly out of scope. The appended OAEI-2025 campaign results (pages 21+) temper the claims: overall $MRR^* \approx 0.65$, recall lags precision, and traditional systems remain "comparable" on simple tasks.

Implementation Feasibility구현 실현가능성
  • Needs: PostgreSQL + pgvector; Python 3.10.12; LangChain (0.2.x) + langchain-openai/-anthropic/-community; rdflib, nltk, tiktoken; Ollama for open-source LLMs; API keys for gpt-4o / Claude; OpenAI embeddings (text-embedding-ada-002, 1536-d). Reference hardware: RTX 4090, 64 GB RAM (for local 7–9B models).
  • Code: available at https://github.com/qzc438/ontology-llm (artifact-evaluated; separate large-dataset repo ontology-llm-large-datasets).
  • Modifications: for our setting, need reference-free operation (present in OAEI-2025 branch), subsumption/complex-relation support (partial, OAEI-2025), and cost control for large/continuous KBs (Agent-OM-Lite). Entity naming normalisation (Type-1/Type-2) must be adapted to the target KB's identifier scheme.
  • Unknowns: missing link — exact F1 numbers not extracted (figures, not tables); missing link — statistical robustness across runs (single-run reporting); missing link — behaviour on schemas ≫ 200 entities beyond biomedical tracks (long runtime reported).

Condition Delta — transfer to our setting
조건 차이 — 우리 환경으로의 전이

Paper's setting: inputs = two OWL/RDFS (later SKOS) ontologies with 100–200 entities each; outputs = 1:1 equivalence alignment between classes/properties (TBox only, no instances); conditions = static ontology pair, gpt-4o backbone, human-curated OAEI reference alignments for evaluation, similarity 0.90 / top@k 3. Our setting (SP7 within dependency-extraction / autonomous KB-maintenance survey): inputs = evolving KB / extracted dependency graph whose new nodes must be linked to existing taxonomy/ontology entities; outputs = alignment links integrated by an autonomous KB-maintenance agent; conditions = continuous, reference-free operation, possibly large/heterogeneous schemas, minimal human intervention. Differences:

  • Static ontology pair vs. continuously-updated KB: reference only — Agent-OM's OAEI-2025 extension adds a reference-free mode (dummy reference then self-replace), which is the missing bridge; verify in deep read.
  • Equivalence-only TBox matching vs. richer dependency/subsumption linking: missing link — subsumption & complex relations explicitly excluded (Limitations §8), though OAEI-2025 §3.2 sketches subsumption support.
  • Agent architecture (planning + tools + shared memory) directly matches "adoptable by an autonomous KB agent": resolved: the two-agent + tool-calling + hybrid-DB pattern is directly transplantable.
  • Cost/latency: LLM-agent OM has high latency and API cost, linear in #entities: missing link for large-scale continuous KB maintenance — mitigated by Agent-OM-Lite (open-source LLMs) but with lower accuracy.
  • Domain generality: evaluated on conference/anatomy/materials-science ontologies: reference only for arbitrary dependency KBs.
