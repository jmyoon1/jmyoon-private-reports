getzep/graphiti - real-time temporal knowledge graphs for AI agents

SP: SP6

ToolTrack -

Problem
해결하려는 문제

Graphiti is an open-source Python framework (28.3k stars) for building and querying temporally-aware knowledge graphs for AI agents. Its defining property: new data ("episodes") integrate into the graph immediately, without batch recomputation — the graph evolves in real-time as episodes arrive, with sub-second query latency. Each fact carries a bi-temporal validity window and source provenance, and when information changes the old fact is invalidated (not deleted), preserving temporal history. Directly demonstrates the incremental, agent-driven KG-update pattern central to O2/O3.

배치 KG 빌더는 새 데이터마다 코퍼스를 재처리해야 하며 이력 질의를 지원하지 못한다.

Solution / Contribution
해결책 / 기여
  1. Invocation: add_episode(...) per new datum (text or structured JSON). Query via

hybrid search methods. Also exposed through an MCP server.

  1. Extraction: LLM with Structured Output support (OpenAI default; Anthropic, Gemini,

Groq, OpenAI-compatible) extracts entities and relationships.

  1. Incremental merge: new nodes/edges are resolved against the existing graph and

merged; no full recompute.

  1. Temporal bookkeeping: contradicting facts trigger automatic invalidation of the

old edge while preserving history via validity windows (bi-temporal).

  1. Output artifacts: a queryable temporal graph persisted in Neo4j, FalkorDB, Amazon

Neptune, or Kuzu.

각 데이터를 episode로 즉시 통합하고 모순 사실을 삭제가 아닌 무효화하는 bi-temporal 유효 창을 갖춘 Graphiti로, 재계산 없이 증분 통합과 시점 질의를 지원한다.

Key Results
주요 결과
  • O1 (dependency/relation extraction): partial — extraction is LLM-driven and generic,

not specialized for dependency structure; ontology can be constrained via Pydantic custom types. reference only for extraction-quality concerns.

  • O2 (incremental KG construction): strong, direct — episode-based immediate

integration without batch recompute is the flagship feature.

  • O3 (temporal / evolving KG maintenance): strong, direct — bi-temporal validity

windows, fact invalidation-not-deletion, point-in-time queries, source provenance.

Concepts Introduced도입된 개념

Episode (ingestion unit)

  • What it computes: raw input (unstructured text or structured JSON) → LLM-driven

entity/edge extraction → new nodes and edges merged into the existing graph → updated temporal graph.

  • Why it was created: batch-oriented KG builders (e.g., GraphRAG) require

reprocessing the corpus when new data arrives; episodes make each new datum an incremental, self-contained update.

  • Concrete example: add_episode(text="Alice moved from Team A to Team B") extracts

entities Alice, Team A, Team B, invalidates the prior Alice—member_of—Team A edge, and adds Alice—member_of—Team B with a new validity window.

  • How it is used: add_episode() is the primary write API; the whole incremental

pipeline hangs off it.

Bi-temporal tracking / validity window

  • What it computes: for each fact, records two time axes — when the fact became

true / ceased to be true (event time) and when the system learned it (ingestion time) → allows point-in-time queries ("what was true as of date X").

  • Why it was created: single-timestamp KGs cannot distinguish "fact is now false"

from "fact was deleted," losing audit/provenance and historical query ability.

  • Concrete example: a fact superseded at t2 still answers a query "as of t1."
  • How it is used: underlies fact invalidation and time-travel queries.

Hybrid retrieval

  • What it computes: query → combines semantic embeddings + BM25 keyword search +

graph traversal → ranked results, without reliance on LLM summarization at query time.

  • Why it was created: LLM-summary-based retrieval (GraphRAG) adds seconds-to-tens-of-

seconds latency; hybrid retrieval targets sub-second, high-precision lookups.

  • Concrete example: README claims "typically sub-second latency" vs GraphRAG's

"seconds to tens of seconds."

  • How it is used: the read side of the framework, complementary to add_episode.

Custom Types (ontology) — developer-defined entity/edge types via Pydantic models, allowing a prescribed ontology or letting structure emerge from data.

5Cs Analysis5Cs 분석

Category: system / open-source framework (production tool, not a research benchmark). Context: positions against Microsoft GraphRAG (static, batch document summarization); backed by the Zep paper "Zep: A Temporal Knowledge Graph Architecture for Agent Memory" (arXiv:2501.13956). Correctness: — README makes latency/accuracy claims ("sub-second") without in-line benchmarks; would need the arXiv paper to verify. High star count (28.3k) is an adoption, not correctness, signal. Contributions: (1) incremental episode integration; (2) bi-temporal fact model with invalidation + provenance; (3) hybrid (embeddings + BM25 + graph) retrieval without LLM-summary dependency. Clarity: API surface (add_episode, hybrid search) is clear; internal dedup/entity- resolution and community-detection details are not in the skimmed section.

Implementation Feasibility구현 실현가능성
  • Needs: Python 3.10+, a supported graph DB (Neo4j / FalkorDB / Amazon Neptune / Kuzu),

an LLM provider with Structured Output (OpenAI/Anthropic/Gemini/Groq/compatible).

  • Code: available at https://github.com/getzep/graphiti (MIT-style OSS, active).
  • Modifications: define dependency-oriented entity/edge types via Pydantic custom

types; validate incremental dedup/entity-resolution behavior on our corpus.

  • Unknowns: missing link — entity deduplication/resolution across episodes and

community-detection specifics not covered in the skimmed README; latency/accuracy claims unbenchmarked here.

Borrow Potential차용 가능성
  • Reusable pattern (high): the episode → immediate incremental merge loop and the

bi-temporal "invalidate, don't delete" fact model with provenance are directly worth extracting for O2/O3 — a clean, battle-tested blueprint for evolving-KG maintenance.

  • Reusable pattern (medium): hybrid retrieval (embeddings + BM25 + graph traversal)

without query-time LLM summarization.

  • Not directly reusable: generic agent-memory extraction; we would need to swap in a

dependency-specific extraction schema (Pydantic custom types as the hook).

Condition Delta — transfer to our setting
조건 차이 — 우리 환경으로의 전이

Tool's setting: inputs = episodes (text or structured JSON) from user interactions / enterprise data; outputs = temporal context graph for agent memory; conditions = LLM with Structured Output + a supported graph DB (Neo4j/FalkorDB/Neptune/Kuzu), Python 3.10+. Our setting: dependency extraction + KG construction over a document corpus, with incremental/evolving updates as the corpus grows (SP6 / O3 focus). Differences:

  • Extraction target: agent-memory entities/facts vs. our dependency relations —

missing link: no dependency-specific extraction schema (mitigable via Pydantic custom types, but unproven for dependencies).

  • Update trigger: per-interaction episodes vs. our document/corpus ingestion —

resolved: add_episode accepts arbitrary text/JSON, so a document = an episode.

  • Temporal model: bi-temporal validity windows — `resolved: directly reusable for

evolving-KG maintenance (O3)`.

  • Retrieval: hybrid search geared to agent recall vs. our analytical queries —

reference only.

  • Entity resolution across episodes — `missing link: dedup mechanism undocumented in

skim`.
