neo4j-labs/llm-graph-builder - Neo4j graph construction from unstructured data via LLMs

SP: SP6

ToolTrack -

Problem
해결하려는 문제

Production web app that transforms unstructured sources (PDF, DOCX, TXT, YouTube transcripts, Wikipedia, web pages) into a Neo4j knowledge graph using an LLM + LangChain extraction pipeline, then serves GraphRAG-style Q&A over the result. Concrete, deployed reference implementation of the exact extract → link → store → query loop the target system needs.

비정형 데이터를 LLM으로 Neo4j 그래프로 구성하고 중복을 해소해야 한다.

Solution / Contribution
해결책 / 기여

Invocation: web UI. Steps:

  1. Connect to a Neo4j 5.23+ instance (APOC plugin required).
  2. Select unstructured sources; optionally define a schema (custom or pre-configured).
  3. Pick an LLM (11 supported: OpenAI, Gemini, Anthropic Claude, Groq, AWS Bedrock, Ollama, Deepseek, Diffbot) and an embedding model (OpenAI, Gemini, Amazon Titan, Sentence Transformers).
  4. "Generate Graph" → documents are chunked, chunks are sent to the LLM for node/relationship/property extraction, and results are written to Neo4j. Token usage is tracked.
  5. Deduplication merges near-duplicate nodes (default score 0.97, text distance 5).
  6. "Chat with Data" answers questions via one of seven retrieval modes; graph viewable in Neo4j Bloom.

Output artifacts: Neo4j KG (nodes, typed relationships, properties) + vector/entity embeddings + full-text indexes.

스키마 하 LLM 추출과 점수화 개체 해소(유사도 0.97+edit-distance 5) dedup을 결합한 Neo4j LLM Graph Builder로 운용 가능한 reuse-vs-create 규칙을 제공하나 acyclicity·cycle 검출은 없다.

Key Results
주요 결과

4,929 GitHub stars (neo4j-labs official). Actively maintained; broad LLM/embedding provider support and a hosted demo indicate production usage.

Concepts Introduced도입된 개념

Tool/repo note — features rather than paper concepts. where the README's first section is insufficient.

LLM-based node/relationship extraction (LangChain LLMGraphTransformer-style)

  • What it computes: unstructured chunk text → LLM prompt → structured (node, relationship, property) triples written to Neo4j. Input = document chunks; operation = LLM extraction under an optional user schema; output = graph nodes + typed edges + properties.
  • Why it was created: to remove the need for hand-built extraction rules / custom trained IE models when turning documents into a KG.
  • Concrete example: a PDF paragraph becomes (:Person {name})-[:WORKS_AT]->(:Company {name}).
  • How it is used here: core ingestion step — "Generate Graph" over selected files or all files in New status.

GraphRAG retrieval modes ("Chat with Data")

  • What it computes: user question → one of seven retrieval strategies over the KG + vector index → grounded answer. Modes: vector, graph_vector, graph, fulltext, graph_vector_fulltext, entity_vector, global_vector.
  • Why it was created: to combine vector similarity with graph traversal so answers are grounded in explicit relationships, not just chunk embeddings.
  • Concrete example: entity_vector mode retrieves via entity embeddings then expands along KG edges.
  • How it is used here: query/consumption layer on top of the constructed KG.

Deduplication / entity resolution (scored merge)

  • What it computes: candidate duplicate nodes → similarity score + text-edit distance → merge decision. Defaults: duplicate score 0.97, text distance 5.
  • Why it was created: to keep the incrementally-built KG from fragmenting into near-duplicate nodes across documents.
  • Concrete example: two nodes "GPT-4" and "GPT 4" within edit distance 5 and score ≥ 0.97 are merged.
  • How it is used here: post-extraction cleanup — directly relevant to O2 (reuse-vs-create) and O3 (redundant-edge avoidance).

Objective Mapping목표 매핑
  • O1 (extract dependency signals): Strong demonstrator of LLM-prompt-based node/relationship extraction from paper-like text under an optional schema — shows the "adoptable via prompting, no custom training" path O1 asks to flag. Does NOT do citation-intent/function classification (SciCite/ACL-ARC style); extraction is generic entity/relation, not citation-lineage-specific. resolved: generic relation extraction | missing link: citation-intent classification.
  • O2 (link vs. create new node): Directly relevant — the scored dedup (similarity 0.97 + edit distance 5) IS an operational reuse-vs-create decision rule, and embedding-model selection supports similarity-threshold linking. Maps to O2's "embedding-similarity thresholds / novelty checks usable by an agent." No formal ontology/taxonomy alignment or prerequisite-matching. resolved: threshold-based entity resolution | missing link: prerequisite/taxonomy alignment.
  • O3 (DAG global consistency): Weakest coverage. Incremental ingestion (process files in New status) + dedup avoids redundant nodes, but no cycle detection, no DAG-acyclicity enforcement, no link-prediction/KG-completion surfaced. missing link: cycle/consistency maintenance.
Implementation Feasibility구현 실현가능성
  • Needs: Neo4j 5.23+ with APOC, Python 3.12+, FastAPI backend, React frontend, an LLM API key + embedding model.
  • Code: available at https://github.com/neo4j-labs/llm-graph-builder (open source, 4.9k stars).
  • Modifications: add citation-intent/function classification for O1; add DAG acyclicity + cycle-detection + link-prediction for O3; wrap the FastAPI backend for autonomous agent invocation instead of manual UI.
  • Unknowns: missing link: exact dedup algorithm internals; whether any consistency/validation hooks exist beyond dedup.
Borrow Potential차용 가능성
  • Adopt as-is: the scored dedup rule (embedding similarity ≥ 0.97 AND edit distance ≤ 5 → merge) as a starting reuse-vs-create heuristic for O2 node-linking; tune thresholds on a concept-linking benchmark.
  • Extract sub-pattern: the seven-mode GraphRAG retrieval taxonomy (esp. entity_vector, graph_vector) as a menu of grounding strategies for the agent's linking/novelty-check step.
  • Reference architecture: overall chunk → LLM-extract-under-schema → write → dedup → GraphRAG loop is a validated blueprint; our pipeline adds a citation-intent extraction stage (O1) and a DAG-consistency/cycle-check stage (O3) that this tool lacks.

  • LangChain LLMGraphTransformer docs — the underlying extraction primitive; check schema-constrained extraction reliability.
Condition Delta — transfer to our setting
조건 차이 — 우리 환경으로의 전이

Tool's setting: general unstructured documents → Neo4j property graph; generic entity/relation extraction; web-UI human-in-the-loop; GraphRAG consumption. Not DAG-constrained, not scientific-citation-specific. Our setting: newly-added paper → dependency-signal extraction (prerequisite concepts + citation lineage) → link to existing KB nodes or spawn new → maintain globally-consistent dependency DAG incrementally, driven by an LLM agent. Differences:

  • Extraction target: generic entities vs. prerequisite-concept + citation-derivation signals — missing link: no citation-intent layer.
  • Graph type: arbitrary property graph vs. constrained DAG — missing link: no acyclicity/consistency layer.
  • Linking: scored dedup merge — resolved: reusable as reuse-vs-create heuristic (embedding sim + edit distance).
  • Driver: web-UI human-in-the-loop vs. autonomous LLM agent — reference only: pipeline is a UI app, not an agent API (though backend FastAPI endpoints are scriptable).
  • Storage/stack: Neo4j 5.23+ + APOC + LangChain — reference only (adoptable but heavyweight for the target).
