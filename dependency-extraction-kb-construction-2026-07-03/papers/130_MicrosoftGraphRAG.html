microsoft/graphrag - modular graph-based Retrieval-Augmented Generation system

SP: SP6

ToolTrack -

Problem
해결하려는 문제

GraphRAG is Microsoft's production-grade reference implementation of an end-to-end LLM knowledge-graph-construction + RAG pipeline: it "extract[s] meaningful, structured data from unstructured text using the power of LLMs," builds an entity knowledge graph, detects communities, pre-generates hierarchical community summaries, and serves graph-augmented retrieval. It is the engineered counterpart to the GraphRAG paper (Edge et al. 2024, SP6 ref, arXiv:2404.16130).

비정형 사설 코퍼스에서 스키마·학습된 NER/RE 없이 지식그래프를 구축할 필요가 있다.

Solution / Contribution
해결책 / 기여

Invocation: graphrag init --root [path] --force to scaffold config, then an index build, then graphrag query. Stages:

  1. Chunk source documents into text units.
  2. Extract entities + relationships (+ descriptions) per chunk via LLM prompt.
  3. Build graph — merge duplicate entities across chunks, aggregate edges.
  4. Community detection — Leiden hierarchical clustering over the graph.
  5. Community summarization — LLM generates a report per community per level.
  6. Serve — Global Search (map-reduce over community summaries) or Local Search

(entity-neighborhood traversal + associated text units).

LLM 개체/관계 추출→Leiden 커뮤니티 탐지→계층적 요약을 결합한 참조 구현 GraphRAG로, 그러나 property graph일 뿐 DAG 불변식·증분 삽입·인용 계보를 제공하지 않는다.

Key Results
주요 결과
  • O1 (extract dependency signals): Directly relevant. Stage 2's LLM

entity/relation extraction is a schema-flexible, prompt-only way to surface candidate concept nodes and typed relations from a new paper's text — exactly the "extract candidate dependency signals" capability. It does NOT do citation-function/intent classification (no reference-list reasoning); it is concept/relation extraction only.

  • O2 (link to existing nodes / spawn new): Partially relevant. Stage 3's

entity resolution (merging duplicate entities emitted across chunks by name + description) is a concrete matching/dedup mechanism. But it is intra-batch merge during a full index build, not incremental linking of a newly-added item against a persistent existing KB.

  • O3 (DAG maintenance / global consistency): Weakly relevant. GraphRAG builds

an undirected/typed property graph, NOT a DAG; it has no cycle avoidance, redundant-edge pruning, or stale-link detection. Its "maintenance" story is coarse: re-index on major version bumps (migration notebook to "avoid re-indexing prior datasets"), graphrag init --force between minor bumps. Community summaries go stale on graph change unless rebuilt.

Concepts Introduced도입된 개념

LLM entity/relation extraction (indexing prompt)

  • What it computes: text chunks → LLM prompt that emits typed entities and

relationships (with descriptions) → node/edge records for a property graph.

  • Why it was created: to build a knowledge graph from unstructured private

corpora without a predefined schema or trained NER/RE model.

  • Concrete example: a chunk about "Dijkstra's algorithm" yields entity nodes

(Dijkstra, shortest-path) and an edge Dijkstra --describes--> shortest-path.

  • How it is used: first stage of the graphrag index pipeline; produces the

base graph that all later stages operate on.

Community detection + hierarchical community summarization

  • What it computes: property graph → Leiden hierarchical clustering → for each

community at each level, an LLM-generated summary report.

  • Why it was created: to answer global/"sensemaking" queries that no single

chunk answers, by pre-computing summaries of graph regions.

  • Concrete example: all nodes about "graph algorithms" cluster into one

community whose summary is retrieved for a global query.

  • How it is used: powers Global Search; local search instead traverses

entity neighborhoods.

Global vs. Local Search retrieval modes

  • What it computes: query → (Global) map-reduce over community summaries, or

(Local) entity-anchored subgraph + linked text units → LLM answer.

  • Why it was created: to separate corpus-wide sensemaking (global) from

entity-specific fact lookup (local).

  • How it is used: the graphrag query command's two primary methods.

Borrow Potential차용 가능성

Reusable sub-patterns for our LLM-agent dependency-DAG pipeline:

  1. Prompt-driven typed entity+relation extraction with descriptions — adopt the

pattern of emitting (entity, type, description) and (src, dst, relation, description) records; the description field is what enables later semantic dedup/linking (O1 → O2 bridge).

  1. Name+description entity resolution as a link-vs-spawn primitive — reuse for

O2: when a new paper's extracted concept matches an existing KB node by name+embedding of description, link; else spawn. Needs augmentation with a persistent-store lookup (GraphRAG merges within a build only).

  1. Hierarchical community summaries — reusable for O3 global-consistency audits:

community-level summaries could support "does this new edge contradict the cluster's known structure" checks, though GraphRAG itself doesn't do this.

NOT borrowable as-is: no acyclicity enforcement, no incremental single-item insert, no citation-lineage extraction — these O3/O1-citation gaps must be sourced elsewhere (e.g., incremental-KG-construction and citation-intent papers in SP6).

Condition Delta — transfer to our setting
조건 차이 — 우리 환경으로의 전이

Tool's target environment: batch indexing of a static unstructured text corpus into a typed entity property graph; retrieval for query-focused summarization. Our target environment: incremental addition of one new paper at a time to a persistent dependency DAG, with prerequisite/derivation edges, cycle-free and globally consistent. Differences:

  • Graph type (property graph vs. directed acyclic prerequisite graph): missing link

— GraphRAG has no acyclicity or edge-direction-consistency machinery.

  • Incremental update (full re-index vs. single-item incremental insert): missing

link — GraphRAG's update model is coarse re-indexing, not append-one-node.

  • Signal source (body-text concepts only vs. concepts + citation-intent lineage):

reference only — GraphRAG ignores the reference list / citation function.

  • Node linking (intra-build entity merge vs. link-to-existing-KB decision):

resolved partially / GraphRAG's name+description entity resolution is a borrowable dedup primitive but not a link-vs-spawn decision policy.
