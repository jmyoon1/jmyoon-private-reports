Tencent/WeKnora - LLM knowledge platform with agent mode and self-maintaining wiki

SP: SP6

ToolTrack -

Problem
해결하려는 문제

WeKnora is an open-source enterprise RAG/agent platform (17.7k stars, Tencent-backed) that turns raw documents into three artifacts: (1) a queryable RAG store, (2) an autonomous ReAct reasoning agent, and (3) a self-maintaining, interlinked markdown wiki with a visual knowledge graph. Relevant to this survey for its Wiki Mode, where agents "distill raw documents into a self-maintaining, interlinked markdown knowledge base" — a shipping instance of autonomous-agent incremental KB construction and maintenance (O2/O3).

Tagline: "Turn Documents into Living Knowledge with RAG, Agents and Auto-Wiki."

자가 유지되는 상호연결 위키를 에이전트로 구축·동기화할 필요가 있다.

Solution / Contribution
해결책 / 기여

Modular, fully swappable pipeline (skim-level; stage internals ):

  1. Ingest / sync — parses 10+ document formats; auto-syncs from Feishu, Notion,

Yuque, and RSS with incremental or full sync options; per-upload process config and batch reparse.

  1. Chunk — adaptive 3-tier configurable chunking.
  2. Vectorize + index — embeddings written across multiple vector DBs.
  3. Retrieve — BM25 / dense / GraphRAG (graph-structured retrieval).
  4. Reason — ReAct Agent progressively orchestrates retrieval, MCP tools, and

web search for multi-step tasks with customizable conversation strategy.

  1. Wiki Mode — agents distill retrieved/ingested docs into an interlinked markdown

KB with a visual knowledge graph, self-maintained as sources change.

Interfaces: Web UI (localhost), RESTful API, CLI (weknora auth login, weknora kb list, weknora chat), Chrome extension, embed widget, WeChat Mini Program. Dependencies: Docker + Docker Compose + Git; optional Neo4j, MinIO, Langfuse via Docker profiles; 20+ LLM provider integrations.

증분 동기화→에이전트 distill→자가 유지 위키를 만드는 WeKnora로 시스템 수준 유지 템플릿을 제공하나 typed 의존 링크·DAG 불변식은 없다.

Key Results
주요 결과

17,722 GitHub stars; Tencent-backed; integrated into the WeChat Dialog Open Platform for zero-code Q&A deployment; 20+ LLM provider integrations; active releases (v0.6.3 adds embed widgets, integrations center, wiki folders). Strong adoption.

Concepts Introduced도입된 개념

Wiki Mode (self-maintaining interlinked KB)

  • What it computes: ingested/parsed documents → agent-driven distillation → an

interlinked markdown knowledge base + visual knowledge graph, refreshed as sources update. Precise link-formation and dedup logic not exposed in README .

  • Why it was created: static RAG stores answer queries but do not produce a

human-browsable, maintained knowledge artifact; Wiki Mode targets a persistent, self-updating KB.

  • Concrete example: connecting a Notion/Feishu space with incremental sync yields a

markdown wiki whose pages interlink and update when the upstream docs change.

  • How it is used here: direct exemplar of O2 (link new material into an existing KB)

and O3 (keep the structure maintained incrementally) as a production feature.

ReAct Agent (progressive multi-step reasoning)

  • What it computes: user query → interleaved reasoning/acting loop calling retrieval,

MCP tools, and web search → multi-step answer. Standard ReAct pattern.

  • Why it was created: single-shot RAG cannot decompose multi-hop tasks; the agent

loop orchestrates tools across steps.

  • Concrete example: a question needing both KB lookup and a web fact triggers

sequential tool calls within one ReAct trajectory.

  • How it is used here: the "autonomous agent" half of the O2/O3 incremental-KB claim.

GraphRAG retrieval — reuses graph-structured RAG (see [82], [81]); here it is one selectable retriever among BM25/dense.

Implementation Feasibility구현 실현가능성
  • Needs: Docker + Docker Compose + Git; optional Neo4j/MinIO/Langfuse; an LLM provider.
  • Code: available at https://github.com/Tencent/WeKnora (Apache-style open source, v0.6.3).
  • Modifications: replace generic wiki-linking with typed dependency extraction/resolution;

add a DAG-consistency module (cycle/redundancy/stale checks) on top of the wiki graph.

  • Unknowns: `missing link: exact Wiki Mode linking algorithm and any consistency checks

not documented in README — requires deep read of source / docs`.

Borrow Potential차용 가능성
  • System pattern (adopt-as-reference): the incremental-sync → agent-distill →

self-maintaining-interlinked-KB loop is a concrete, battle-tested skeleton for the O2/O3 maintenance layer of an agent-driven KB. Worth extracting as an architectural template.

  • Not borrowable as-is: no typed dependency extraction (O1) and no DAG-consistency

primitives (O3); those must be built on top.

  • Sub-pattern: per-upload process config + batch reparse as the hook where a

dependency-extraction step (O1) and a link-resolution step (O2) could be injected.

Condition Delta — transfer to our setting
조건 차이 — 우리 환경으로의 전이

Tool's setting: inputs = enterprise documents (10+ formats) from Feishu/Notion/Yuque/RSS; outputs = RAG answers, ReAct agent responses, self-maintaining markdown wiki + visual KG; conditions = Docker-deployed platform, relatedness/semantic linking, doc-level maintenance. Our setting: inputs = newly-added academic papers + their citations; outputs = typed prerequisite/derivation edges into a dependency DAG of KB nodes; conditions = LLM-agent pipeline, incremental global DAG consistency (cycle-free, pruned, stale-link-detected). Differences:

  • Link semantics: WeKnora links = generic relatedness/wiki interlink; ours = typed

prerequisite/derivation dependency → missing link: no typed dependency extraction (O1).

  • Structure invariants: WeKnora maintains doc/wiki freshness; ours needs DAG invariants

(acyclicity, redundant-edge pruning) → missing link: no DAG-consistency layer (O3).

  • Node granularity: WeKnora = documents/wiki pages; ours = concept/paper nodes → reference only.
  • Incremental sync + self-maintaining wiki architecture → `resolved: reference architecture

for agent-driven incremental KB maintenance we can borrow at the system level`.

  • Domain: enterprise docs vs. scientific papers → reference only.
